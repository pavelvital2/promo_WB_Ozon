# 05_ACCESS_CONTROL_MODEL.md

## 1. Назначение документа

Документ фиксирует:
- модель доступа системы;
- взаимодействие ролей, permissions и store access;
- правила допуска к действиям, страницам и данным;
- правила скрытия разделов интерфейса;
- правила доступа к history, logs, run pages и file download;
- поведение пользователя без магазинов;
- обязательные инварианты access enforcement.

Документ не описывает:
- структуру БД;
- маршруты UI;
- жизненный цикл runs;
- логику WB/Ozon.

---

## 2. Краткий итог

В системе действует трёхкомпонентная модель доступа:

1. базовая роль пользователя;
2. отдельные explicit permissions;
3. отдельная привязка пользователя к магазинам.

Проверка доступа никогда не строится только на одном уровне.
Для большинства прикладных действий требуется комбинация:
- роль;
- permission, если действие permission-зависимое;
- store access, если действие связано с конкретным магазином или его данными.

Администратор имеет полный доступ ко всей системе.
Управляющий и Менеджер ограничены только назначенными магазинами.
Управляющий может получить ограниченные права на создание/редактирование магазинов через permissions.
Менеджер не управляет пользователями, магазинами и настройками, не видит системные логи.

Недоступные разделы интерфейса скрываются полностью.
Они не показываются как disabled и не сопровождаются сообщением “нет прав”.
Исключение — формально доступный раздел без доступных магазинов: в этом случае показывается пустое состояние “Нет доступных магазинов”.

## 3. Компоненты модели доступа

## 3.1. Роль

Роль задаёт базовый уровень системных возможностей пользователя.

Обязательные роли:
- admin
- manager_lead
- manager

Роль определяет:
- верхнюю границу допустимых действий;
- набор потенциально доступных разделов;
- право доступа к пользователям, логам, магазинам, настройкам.

Роль не определяет сама по себе:
- доступ к конкретному магазину;
- наличие дополнительных прав на create_store/edit_store.

## 3.2. Permission

Permission — это точечное дополнительное право, которое может расширять возможности пользователя в рамках, разрешённых ТЗ.

Обязательные permission:
- create_store
- edit_store

Permission:
- назначается пользователю явно;
- не заменяет роль;
- не даёт доступа к магазинам сам по себе;
- всегда применяется в сочетании с ролевой моделью и store access.

## 3.3. User store access

User store access — это явная привязка пользователя к конкретным магазинам.

Она определяет:
- какие магазины пользователь видит;
- по каким магазинам пользователь может выполнять check/process;
- какие history rows он может видеть;
- какие run pages и файлы он может открывать;
- какие store-specific данные он может читать.

Для Управляющего и Менеджера доступ к store-bound данным без user_store_access запрещён.

## 4. Общий алгоритм проверки доступа

Для любого действия прикладной слой обязан выполнять проверку в следующей логике:

1. Пользователь авторизован.
2. Пользователь не заблокирован.
3. Действие допустимо по роли.
4. Если для действия нужен permission:
   - проверяется наличие permission.
5. Если действие привязано к магазину:
   - проверяется store access либо полное администраторское право.
6. Если магазин архивирован:
   - применяются дополнительные ограничения сценария.
7. Только после этого действие выполняется.

Это правило обязательно для:
- page access;
- action access;
- data access;
- file download;
- run creation;
- logs access.

## 5. Роли и базовые возможности

## 5.1. Администратор

Базовые права:
- полный доступ ко всей системе;
- создание, редактирование, блокировка и разблокировка пользователей;
- назначение ролей;
- назначение permissions;
- создание, редактирование, архивирование и восстановление магазинов;
- назначение пользователей на магазины;
- просмотр всех запусков, файлов, аудитов и логов;
- изменение настроек всех магазинов.

Дополнительно:
- для Администратора не требуется user_store_access для доступа к магазину;
- отсутствие записи в user_store_access не ограничивает Администратора.

UI-доступ:
- логин
- дашборд
- пользователи
- магазины
- обработка Wildberries
- обработка Ozon
- история запусков
- логи
- страница run/detail audit
- смена собственного пароля

## 5.2. Управляющий

Базовые права:
- доступ только к назначенным магазинам;
- загрузка, check, process, просмотр аудита и скачивание результатов только по назначенным магазинам;
- просмотр history только по назначенным магазинам.

Дополнительные возможные права:
- при наличии `create_store` может создавать магазины любого маркетплейса;
- созданный им магазин автоматически попадает в его доступный контур;
- при наличии `edit_store` может редактировать только те магазины, к которым привязан;
- может менять настройки Wildberries только если:
  - имеет `edit_store`
  - и магазин ему доступен.

Явные запреты:
- не назначает пользователей на магазины;
- не архивирует магазины;
- не восстанавливает магазины;
- не редактирует непривязанные магазины;
- не управляет пользователями;
- не видит раздел пользователей;
- не получает полного доступа ко всем магазинам;
- не получает права на логи по умолчанию и по permission.

UI-доступ:
- логин
- дашборд
- обработка Wildberries — только если есть доступные WB магазины
- обработка Ozon — только если есть доступные Ozon магазины
- история запусков — только если есть доступные магазины
- страница run/detail audit — только для доступных ему runs
- смена собственного пароля
- раздел “Магазины” — только если у него есть хотя бы одно магазинное permission:
  - create_store
  - edit_store

Не видит:
- пользователи
- логи

## 5.3. Менеджер

Базовые права:
- доступ только к назначенным магазинам;
- загрузка, check, process, просмотр аудита и скачивание результатов только по назначенным магазинам;
- просмотр history только по назначенным магазинам.

Явные запреты:
- без права управления пользователями;
- без права управления магазинами;
- без права изменения настроек магазинов;
- без доступа к системным логам;
- не видит раздел пользователей;
- не видит раздел магазинов.

UI-доступ:
- логин
- дашборд
- обработка Wildberries — только если есть доступные WB магазины
- обработка Ozon — только если есть доступные Ozon магазины
- история запусков — только если есть доступные магазины
- страница run/detail audit — только для доступных ему runs
- смена собственного пароля

Не видит:
- пользователи
- магазины
- логи

## 6. Модель сочетания роль + permission + store access

## 6.1. Общий принцип

Для действия над магазином или его данными недостаточно только роли.
Должны одновременно выполняться все требуемые условия.

Типовые сочетания:

### Пример 1. Запуск check/process менеджером
Нужно:
- роль manager
- store access к конкретному магазину
- магазин active
- модуль соответствует магазину
- нет active run conflict

### Пример 2. Редактирование магазина управляющим
Нужно:
- роль manager_lead
- permission edit_store
- store access к магазину
- магазин active

### Пример 3. Создание магазина управляющим
Нужно:
- роль manager_lead
- permission create_store

Store access к создаваемому магазину заранее не нужен, потому что:
- после создания магазин автоматически добавляется в его доступный контур.

### Пример 4. Просмотр логов
Нужно:
- только роль admin

Никакой permission не расширяет logs access для manager_lead или manager.

## 7. Матрица доступа по разделам интерфейса

## 7.1. Страница логина

Видят:
- все неавторизованные пользователи

## 7.2. Дашборд

Видят:
- admin
- manager_lead
- manager

Условия:
- авторизован
- не заблокирован

Особое состояние:
- пользователь без магазинов видит дашборд и сообщение “Нет доступных магазинов”.

## 7.3. Раздел “Пользователи”

Видит:
- только admin

Не видят:
- manager_lead
- manager

## 7.4. Раздел “Магазины”

Видит:
- admin
- manager_lead только если у него есть хотя бы одно право:
  - create_store
  - edit_store

Не видит:
- manager

Дополнительные ограничения для manager_lead:
- список и действия в разделе ограничены его доступным контуром магазинов, кроме самого сценария создания магазина при наличии create_store.

## 7.5. Страница обработки Wildberries

Видят:
- admin
- manager_lead
- manager

Но фактически usable только если:
- у пользователя есть хотя бы один доступный active store с marketplace=wb

Иначе:
- переход в раздел может быть скрыт;
- при прямом открытии допустимо пустое состояние “Нет доступных магазинов”, только если раздел формально доступен роли.

## 7.6. Страница обработки Ozon

Аналогично Wildberries:
- admin
- manager_lead
- manager

Фактически usable только если:
- есть хотя бы один доступный active store с marketplace=ozon

## 7.7. История запусков

Видят:
- admin — все runs
- manager_lead — только runs своих доступных магазинов
- manager — только runs своих доступных магазинов

Если у пользователя нет доступных магазинов:
- раздел истории не показывается.

## 7.8. Страница логов

Видит:
- только admin

Не видят:
- manager_lead
- manager

## 7.9. Страница run / full detail audit

Видят:
- admin — любой run
- manager_lead — только run своего доступного магазина
- manager — только run своего доступного магазина

Условие:
- доступ определяется через принадлежность run к store и право на этот store.

## 7.10. Смена собственного пароля

Видят:
- все авторизованные и не заблокированные пользователи

## 9. Правила store-bound доступа

## 9.1. Общий принцип

Если страница, действие, run, файл, аудит или запись истории принадлежат магазину, то доступ определяется по store-bound правилу.

Store-bound доступ означает:
- для admin доступ разрешён всегда;
- для manager_lead/manager требуется запись в user_store_access для данного store.

## 9.2. Объекты, считающиеся store-bound

Store-bound считаются:
- store page record
- temporary uploaded files
- run
- run_files
- run_summary_audit
- run_detail_audit
- history row
- downloadable source/result files
- system_logs, если доступ к логам был бы разрешён по роли

## 9.3. Derived access

Если у пользователя есть доступ к store, то он может получить доступ к производным объектам этого store:
- runs этого store
- files этих runs
- audits этих runs
- history rows этих runs

Но только в пределах роли.
Например:
- manager всё равно не получает доступ к logs, даже если лог связан с его store.

## 10. Правила доступа к архивированным магазинам

## 10.1. Общий принцип

Архивированный магазин остаётся видимым в истории и в списках магазинов по правилам роли, но не может использоваться в новых запусках и не может редактироваться.

## 10.2. Для администратора

Admin может:
- видеть архивированный магазин;
- видеть его history;
- видеть runs и audits;
- восстанавливать магазин.

Admin не должен запускать новые операции по archived store.

## 10.3. Для управляющего

Manager_lead может:
- видеть архивированный магазин, если он входит в его доступный контур и раздел магазинов ему вообще доступен;
- видеть history runs по нему;
- видеть run pages и файлы по историческим запускам при доступности файлов.

Manager_lead не может:
- редактировать archived store;
- запускать новые операции;
- восстанавливать его;
- архивировать его.

## 10.4. Для менеджера

Manager может:
- видеть history runs по архивированному магазину, если он входит в его доступный контур;
- видеть run pages и исторические файлы при доступности файлов.

Manager не видит раздел магазинов и не управляет archived store.

## 11. Правила видимости разделов в UI

## 11.1. Базовое правило

Недоступные разделы:
- не показываются;
- не показываются disabled;
- не сопровождаются сообщением “нет прав”.

## 11.2. Что считается недоступным разделом

Недоступным разделом считается раздел, к которому пользователь не имеет права по роли/permissions.

Примеры:
- users для manager_lead/manager
- logs для manager_lead/manager
- stores для manager
- stores для manager_lead без create_store и без edit_store

## 11.3. Исключение — пустое состояние при формальном доступе

Если раздел доступен по роли/permissions, но внутри нет ни одного доступного магазина, то вместо скрытия раздела допускается пустое состояние:
- “Нет доступных магазинов”

Это правило относится к:
- dashboard
- processing pages
- другим store-bound разделам, формально доступным роли

## 11.4. История запусков при отсутствии магазинов

Пользователь без доступных магазинов:
- не видит истории запусков;
- не получает переход в модуль history.

Это отдельное правило ТЗ и его нужно соблюдать строже, чем общее пустое состояние.

---

## 12. Пользователь без магазинов

## 12.1. Общие правила

Пользователь без доступных магазинов:
- может войти в систему;
- видит дашборд;
- видит сообщение “Нет доступных магазинов”;
- не видит history;
- не может запускать check/process;
- не видит переходы в недоступные модули обработки.

## 12.2. Особый случай — управляющий с create_store

Если пользователь:
- имеет роль manager_lead
- не имеет магазинов
- но имеет permission `create_store`

то он:
- может войти;
- видит дашборд;
- видит “Нет доступных магазинов”;
- не видит history;
- не может запускать check/process;
- видит возможность создания магазина;
- после создания магазина получает к нему доступ автоматически.

## 13. Access control для history

## 13.1. Область данных

History page должна показывать только runs.

## 13.2. Data scope rules

### Admin
Видит:
- все runs всех stores

### Manager_lead
Видит:
- только runs, где `runs.store_id` принадлежит множеству его доступных stores

### Manager
Видит:
- только runs, где `runs.store_id` принадлежит множеству его доступных stores

## 13.3. Search/filter/sort enforcement

Даже если пользователь пытается:
- отфильтровать по чужому store;
- найти чужой run по номеру;
- искать по username другого магазина;

система обязана сначала ограничить выборку его доступным контуром, а уже потом применять search/filter/sort.

То есть access filter первичен, UI filter вторичен.

---

## 14. Access control для logs

## 14.1. Role rule

Logs доступны только admin.

## 14.2. Consequence

Для manager_lead и manager:
- раздел logs скрыт;
- прямой URL доступ запрещён;
- API/endpoint доступ запрещён;
- никакой permission не должен открывать logs.

---

## 15. Access control для run page и detail audit

## 15.1. Admin

Имеет доступ к любому run page:
- summary audit
- detail audit
- input/output files при их доступности

## 15.2. Manager_lead / Manager

Имеют доступ только если:
- run принадлежит store, к которому у пользователя есть доступ

Доступ включает:
- просмотр run metadata
- просмотр summary audit
- просмотр detail audit
- скачивание доступных файлов run

Если run принадлежит чужому store:
- прямой доступ запрещён;
- run page не должен раскрываться частично;
- возвращается access denied / not found по принятому проектному контракту безопасности.

---

## 16. Access control для file download

## 16.1. Общий принцип

Скачивание файла разрешено только пользователю, имеющему право на соответствующий магазин и запуск.

Проверка всегда включает:
1. авторизацию;
2. не blocked user;
3. право доступа к store/run;
4. наличие файла;
5. `is_available=true`.

## 16.2. Какие файлы защищаются

Защищаются:
- source files в history
- result files
- любые run files
- временные файлы, если предусмотрено их скачивание из UI текущего активного набора

## 16.3. Прямой URL доступ

Прямой публичный доступ к файлам запрещён.
Скачивание всегда идёт через защищённый backend action/use case.

## 17. Access control для create_store и edit_store

## 17.1. Create store

### Admin
- всегда может создавать магазины

### Manager_lead
- может создавать магазин только при наличии `create_store`

### Manager
- не может создавать магазин никогда

## 17.2. Edit store

### Admin
- может редактировать любой active store

### Manager_lead
- может редактировать только если:
  - есть `edit_store`
  - store доступен ему
  - store active

### Manager
- не может редактировать store никогда

## 17.3. Wildberries settings

Изменение настроек WB является частью редактирования store.
Следовательно:
- admin — да
- manager_lead — только при `edit_store` и доступе к store
- manager — нет

---

## 18. Access control для create/update/block users

Все user-management действия доступны только admin.

Никакие permissions:
- create_store
- edit_store

не должны влиять на user-management.

Это жёсткий инвариант.

---

## 19. Enforcement layers

Проверка доступа должна выполняться минимум на двух уровнях.

## 19.1. UI visibility layer

Назначение:
- скрыть недоступные разделы;
- скрыть недоступные действия;
- не показывать пользователю невозможные сценарии.

Но UI visibility не является достаточной защитой.

## 19.2. Application authorization layer

Назначение:
- окончательно проверять право на действие;
- контролировать прямые URL;
- контролировать прямые POST/GET запросы;
- ограничивать read-side выборки по доступному контуру.

Это обязательный уровень защиты.

## 19.3. Repository/query scope layer

Для history/logs/run detail queries доступ должен усиливаться на read-side:
- сначала ограничение по доступным store;
- затем search/filter/sort/page.

Это особенно важно для:
- history
- run page
- file download

## 20. Обязательные deny-сценарии

Система должна запрещать как минимум следующие сценарии:

1. Manager открывает users page.
2. Manager открывает stores page.
3. Manager_lead открывает users page.
4. Manager_lead или Manager открывает logs page.
5. Manager запускает run по чужому store.
6. Manager_lead редактирует чужой store при наличии edit_store.
7. Manager_lead архивирует store.
8. Manager_lead восстанавливает archived store.
9. Любой не-admin назначает user_store_access.
10. Любой не-admin меняет роли пользователей.
11. Пользователь скачивает файл чужого run.
12. Пользователь запускает run по archived store.
13. Blocked user выполняет любой авторизованный сценарий.
14. Пользователь видит hidden sections через menu/UI.

---

## 21. Обязательные allow-сценарии

Система должна разрешать как минимум следующие сценарии:

1. Admin видит все разделы и все данные.
2. Manager_lead с create_store и без магазинов может создать первый магазин.
3. Созданный manager_lead магазин автоматически становится ему доступен.
4. Manager_lead с edit_store может редактировать свой active store.
5. Manager с доступом к store может загрузить файлы и запустить check/process.
6. Manager/manager_lead может открыть history только по своим stores.
7. Manager/manager_lead может открыть run page по своему store.
8. Любой авторизованный неблокированный пользователь может сменить собственный пароль.
9. Пользователь без магазинов может войти и увидеть dashboard с пустым состоянием.
