ТЕХНИЧЕСКОЕ ЗАДАНИЕ
ВАРИАНТ 3
на разработку веб-программы расчёта скидок для Wildberries и Ozon

1. ОБЩИЕ ПОЛОЖЕНИЯ

1.1. Наименование системы

Веб-программа расчёта скидок для Wildberries и Ozon.

1.2. Назначение системы

Система предназначена для:
- загрузки Excel-файлов формата .xlsx;
- валидации структуры и содержимого файлов;
- выполнения отдельной операции проверки;
- выполнения отдельной операции обработки;
- расчёта значений по строго заданной логике для Wildberries и Ozon;
- формирования итогового Excel-файла;
- отображения краткого результата, сводного аудита и детального аудита;
- хранения ограниченной истории запусков и файлов;
- разграничения доступа пользователей по ролям, permissions и магазинам;
- журналирования действий пользователей и системных событий.

1.3. Принцип разработки

Система должна быть реализована строго по данному ТЗ.
Любые допущения, трактовки, альтернативные сценарии и предположения, не описанные в ТЗ, запрещены.
Если в процессе проектирования или реализации обнаружится отсутствие данных, это считается дефектом ТЗ, а не основанием для самостоятельного выбора логики.

1.4. Среда выполнения

- ОС: Ubuntu VPS
- Backend: Python
- База данных: PostgreSQL
- Веб-сервер: nginx
- Запуск приложения: systemd service
- Docker: не используется

1.5. Формат работы системы

Система работает только через веб-интерфейс.
Обработка выполняется только по загруженным пользователем файлам.
Интеграции с API маркетплейсов в MVP отсутствуют.
Автоматические фоновые обработки по расписанию, уведомления, внешние интеграции и аналитические модули в MVP отсутствуют.

1.6. Термины

- Запуск — одна операция типа check или process для одного магазина и одного набора файлов.
- Проверка / check — отдельная самостоятельная операция валидации набора файлов без формирования результирующего файла.
- Обработка / process — отдельная самостоятельная операция расчёта и формирования результирующего файла.
- Магазин — логическая сущность системы, к которой привязаны маркетплейс, настройки расчёта и доступы.
- Источник — загруженный пользователем исходный файл.
- Результат — сформированный системой выходной файл.
- Краткий результат — короткая человекочитаемая строка для списка запусков и дашборда.
- Сводный аудит — агрегированный набор итоговых показателей запуска.
- Детальный аудит — построчная детализация проверки или обработки.
- Критическая ошибка — ошибка, блокирующая успешное завершение операции.
- Предупреждение — отклонение, не блокирующее запуск целиком, но влияющее на отдельные строки, файлы или итоговый результат.

2. ГРАНИЦЫ MVP

2.1. В MVP входит

- авторизация пользователей;
- роли;
- permissions;
- разграничение доступа по магазинам;
- создание, редактирование, архивирование и восстановление магазинов;
- хранение настроек расчёта магазинов;
- загрузка исходных .xlsx-файлов;
- отдельная проверка файлов;
- отдельная обработка файлов;
- обработка Wildberries;
- обработка Ozon;
- скачивание результатов;
- скачивание исходных файлов из истории в пределах срока хранения;
- краткий результат;
- сводный аудит;
- детальный аудит;
- история запусков;
- журнал логов;
- поиск, фильтры, сортировки и пагинация в истории, логах и детальном аудите;
- создание первого Администратора через CLI-команду;
- смена собственного пароля авторизованным пользователем.

2.2. В MVP не входит

- любые API маркетплейсов;
- API-ключи маркетплейсов;
- автоматические периодические запуски;
- уведомления;
- телеграм-боты;
- аналитика;
- массовые очереди обработки для разных подсистем;
- интеграции с внешними сервисами;
- восстановление пароля по email;
- принудительная смена пароля;
- сброс пароля пользователем без участия Администратора;
- поддержка форматов кроме .xlsx;
- экспорт детального аудита отдельным файлом.

2.3. Следствие для проектирования

В интерфейсе MVP не должно быть сущностей, страниц и сценариев управления API-ключами.
Проектировщику запрещено закладывать внешние интеграции и расширять область работ MVP.

4. РОЛИ, ДОСТУПЫ И ПРАВА

4.1. Роли

- Администратор
- Управляющий
- Менеджер

4.2. Модель прав

В системе используется модель:
- базовая роль пользователя;
- отдельные permissions;
- отдельная привязка пользователя к магазинам.

4.3. Администратор

Права:
- полный доступ ко всей системе;
- создание, редактирование, блокировка и разблокировка пользователей;
- назначение ролей;
- назначение permissions;
- создание, редактирование, архивирование и восстановление магазинов;
- назначение пользователей на магазины;
- просмотр всех запусков, файлов, аудитов и логов;
- изменение настроек всех магазинов;
- создание первого администратора через CLI/seed не относится к веб-правам.

4.4. Управляющий

Права:
- доступ только к назначенным магазинам;
- загрузка, проверка, обработка, просмотр аудита и скачивание результатов только по назначенным магазинам;
- просмотр истории запусков только по назначенным магазинам;
- при наличии permission create_store может создавать магазины любого маркетплейса;
- созданный им магазин автоматически попадает в его доступный контур;
- при наличии permission edit_store может редактировать только те магазины, к которым привязан;
- может менять настройки Wildberries только если имеет permission edit_store и магазин ему доступен.

4.5. Менеджер

Права:
- доступ только к назначенным магазинам;
- загрузка, проверка, обработка, просмотр аудита и скачивание результатов только по назначенным магазинам;
- просмотр истории запусков только по назначенным магазинам;
- без права управления пользователями;
- без права управления магазинами;
- без права изменения настроек магазинов;
- без доступа к системным логам.

4.6. Явные запреты для Управляющего и Менеджера

Управляющему запрещено:
- назначать пользователей на магазины;
- архивировать магазины;
- восстанавливать магазины;
- редактировать непривязанные магазины.

Менеджеру запрещено:
- видеть раздел пользователей;
- редактировать магазины;
- менять настройки;
- видеть страницу логов.

4.7. Отображение недоступных разделов

Единое правило для интерфейса:
- разделы, недоступные пользователю по роли/permissions, полностью скрываются;
- не показываются как disabled;
- не показываются с сообщением «нет прав».

Исключение:
- для страницы/раздела, доступ к которому есть формально, но нет доступных магазинов, показывается пустое состояние с сообщением «Нет доступных магазинов».

4.8. Пользователь без магазинов

Пользователь без доступных магазинов:
- может войти в систему;
- видит дашборд;
- видит сообщение «Нет доступных магазинов»;
- не видит истории запусков;
- не может запускать check/process;
- не видит переходы в недоступные модули обработки;
- если это Управляющий с permission create_store, он должен видеть возможность создания магазина.

4.9. Отображение разделов

- Менеджер раздел «Магазины» не видит;
- Управляющий раздел «Пользователи» не видит;
- Администратор видит все разделы;
- Управляющий видит раздел «Магазины» только если у него есть хотя бы одно право, связанное с магазинами: create_store или edit_store.

5. МАГАЗИНЫ

5.1. Обязательные поля магазина

- название магазина
- маркетплейс
- статус

5.2. Уникальность

Название магазина должно быть уникальным в пределах маркетплейса.
Одинаковые названия для разных маркетплейсов допустимы.

5.3. Статусы магазина

- active
- archived

5.4. Архивированный магазин

Архивированный магазин:
- не может использоваться в новых запусках;
- не может редактироваться;
- показывается в истории и в списке магазинов;
- отсутствует в списке выбора для новых запусков;
- может быть восстановлен только Администратором.

5.5. Отображение магазина в интерфейсе

По умолчанию:
- название + маркетплейс + статус

Пример:
- VitalEmb — Wildberries — Активен
- VitalEmb — Ozon — Архивирован

6. ЖИЗНЕННЫЙ ЦИКЛ CHECK И PROCESS

6.1. Общая модель

check и process являются самостоятельными операциями.
Каждая операция создает отдельную запись в runs и отдельную запись в истории.

6.2. Проверка / check

- check не формирует результирующий файл;
- check формирует сводный и детальный аудит;
- check использует отдельный business_result:
  - check_passed
  - check_passed_with_warnings
  - check_failed

6.3. Обработка / process

- process может запускаться без предварительного ручного check;
- перед process всегда выполняется автоматическая валидация;
- автоматическая валидация отражается:
  - как lifecycle_status = validating;
  - как вложенный отдельный блок в карточке process-запуска;
- автоматическая валидация не создает отдельный самостоятельный run.

6.4. Набор статусов process

Lifecycle_status для process:
- created
- validating
- processing
- completed
- failed

Business_result для process:
- validation_failed
- completed
- completed_with_warnings
- failed

6.5. Check-аудит и process-аудит

6.5.1. Общие правила

И check, и process обязаны формировать:
- сводный аудит;
- детальный аудит.

Хранение check-аудита и process-аудита выполняется одинаковым способом.

6.5.2. Граница check-аудита для Wildberries

На этапе check для Wildberries разрешено и требуется:
- проверять структуру файла цен;
- проверять структуру файлов акций;
- нормализовать данные;
- выявлять невалидные строки;
- определять наличие артикула в акциях;
- считать min_discount;
- считать max_plan_price;
- считать calculated_discount;
- считать итоговое значение final_discount_pre_threshold;
- определять, превысился бы threshold;
- определять, применился бы fallback_no_promo;
- определять, применился бы fallback_over_threshold;
- формировать краткий результат, сводный аудит и детальный аудит так, как если бы расчёт был выполнен;
- но не записывать изменения в файл результата.

6.5.3. Граница check-аудита для Ozon

На этапе check для Ozon разрешено и требуется:
- проверять структуру книги;
- проверять обязательный лист и обязательные колонки;
- нормализовать значения;
- по каждой строке вычислять, попала бы она под правило:
  - missing_min_price
  - no_stock
  - no_boost_prices
  - use_max_boost_price
  - use_min_price
  - below_min_price_threshold
  - insufficient_ozon_input_data
- формировать краткий результат, сводный аудит и детальный аудит так, как если бы обработка была выполнена;
- но не записывать изменения в результирующий файл.

7. ВРЕМЕННЫЕ ФАЙЛЫ И НАБОРЫ ФАЙЛОВ

7.1. Что считается изменением состава файлов после check

Любое из перечисленного делает предыдущий check недействительным:
- удаление одного файла;
- добавление одного файла;
- замена файла другим;
- повторная загрузка файла с тем же именем, но иным содержимым;
- повторная загрузка файла с тем же содержимым, но иным именем;
- повторная загрузка файла с тем же именем и тем же содержимым, если он загружен заново и получил новый временный объект.

7.2. Признак идентичности файла

Файл считается «тем же самым» только по комбинации:
- SHA-256 хэш содержимого;
- размер файла в байтах.

Оригинальное имя файла не является достаточным признаком идентичности.

7.3. Активный набор файлов

check/process всегда работают только по текущему активному набору файлов на экране.
Выбор ранее загруженного набора файлов из нескольких наборов в MVP не поддерживается.

7.4. Временные файлы до запуска

На страницах обработки должен отображаться текущий активный набор временно загруженных файлов с возможностью:
- удалить файл из набора;
- заменить файл;
- увидеть имя файла, размер и время загрузки.

7.5. Очистка временных файлов

- временные файлы, не использованные для запуска, должны автоматически очищаться;
- срок хранения временного файла — 24 часа с момента загрузки;
- очистка должна журналироваться как системное событие;
- отдельная UI-индикация факта автоматической очистки не требуется;
- если пользователь открывает экран после очистки, набор должен считаться пустым.

8. ХРАНЕНИЕ И ФИЗИЧЕСКАЯ ОРГАНИЗАЦИЯ ФАЙЛОВ

8.1. Место хранения

Файлы должны храниться в файловой системе VPS в выделенном каталоге приложения.

8.2. Структура каталогов

Минимально обязательная логическая структура:
- /storage/uploads/tmp/
- /storage/runs/{module_code}/{store_id}/{public_run_number}/input/
- /storage/runs/{module_code}/{store_id}/{public_run_number}/output/

Проектировщик вправе уточнить физическую структуру вложенных каталогов, если она не меняет данную логику.

8.3. Отдельные физические копии

Для каждого run входные файлы должны сохраняться как отдельные физические копии.
Дедупликация файлов по хэшу между разными runs в MVP не допускается.

8.4. Метаданные файлов

Для каждого файла обязательно хранить:
- original_filename
- stored_filename
- storage_relative_path
- mime_type
- file_size_bytes
- file_sha256
- uploaded_at_utc
- expires_at_utc
- доступность / недоступность файла

8.5. Поведение UI при недоступном файле

Если срок хранения файла истёк или файл удалён по правилам хранения:
- в карточке запуска и истории должна отображаться надпись «Файл недоступен»;
- кнопка скачивания должна быть disabled;
- у файла должен отображаться отдельный признак недоступности.

8.6. История удаления предыдущего успешного результата

При удалении старого успешного результата из-за появления нового успешного результата система обязана:
- записать системный лог;
- пометить старый файл результата как недоступный;
- сохранить историческую запись запуска.

9. КОНКУРЕНТНОСТЬ И БЛОКИРОВКИ

9.1. Параллельные process-запуски по одному магазину и модулю

Запрещены.

9.2. Параллельные check-запуски по одному магазину и модулю

Разрешены только если не выполняется process по той же паре магазин + модуль.
Но в MVP рекомендуется сериализация всех запусков по паре магазин + модуль.
Итоговое обязательное правило:
- одновременно не допускаются два активных run по одной паре магазин + модуль.

9.3. Одновременные check и process по одному магазину и модулю

Запрещены.

9.4. Одновременные process по разным модулям

Маркетплейс магазина жёстко определяет модуль:
- для магазина WB существует только модуль WB;
- для магазина Ozon существует только модуль Ozon.

Следовательно, пара «магазин + модуль» фактически уникальна по самому магазину.

9.5. Уровень блокировки

Блокировка устанавливается на пару:
- store_id + module_code

9.6. Разрешение гонки для статуса текущего успешного результата

Так как одновременные активные runs по одной паре store_id + module_code запрещены, гонка двух одновременных успешных process-запусков в корректной работе не допускается.
Если по технической ошибке такая ситуация всё же возникла, текущим успешным результатом должен считаться тот, у которого later finished_at_utc.
Другой результат должен быть помечен как superseded, а событие должно быть записано в system_logs как ошибка согласованности.

9.7. Повторный запуск по тем же файлам

Повторный process по тем же файлам разрешён всегда.
Запрета на повторный запуск по идентичному набору файлов нет.

10. ВЫПОЛНЕНИЕ ОПЕРАЦИЙ И ТЕХНИЧЕСКИЙ КОНТРАКТ

Интерфейс обязан показывать:

10.4. Максимальный размер одного файла

10.5. Максимальный суммарный объём файлов на один запуск Wildberries

10.6. Максимальное количество файлов акций Wildberries

13. РАБОТА С EXCEL

13.1. Общие ограничения

Поддерживаются только валидные .xlsx файлы без макросов и без необходимости специальной обработки внешних связей.

13.2. Файлы с макросами, защищённой книгой и защищёнными листами

Файлы считаются невалидными, если:
- книга защищена так, что приложение не может гарантированно прочитать/изменить требуемые данные;
- лист защищён так, что приложение не может записать разрешённые изменения;
- книга имеет макросы/структуры, которые приложение не гарантирует корректно сохранить.

Для MVP такие файлы должны отклоняться как validation_failed или check_failed с явной причиной.

13.3. Внешние связи Excel

Файлы с внешними связями не считаются отдельным поддерживаемым сценарием.
Если книга читается и сохраняется без нарушения обязательных требований — она допустима.
Если внешние связи создают риск некорректного сохранения — файл считается невалидным.

13.4. Контракт Ozon

13.4.1. Определение обязательных колонок

Для Ozon обязательные колонки определяются строго по фактическим буквенным позициям листа:
- J
- K
- L
- O
- P
- R

Заголовки строк 1–3 используются как дополнительная проверка корректности шаблона, но первичным критерием является фактическая позиция колонок.

13.4.2. Нестандартное форматирование в строках 1–3

Merged cells, нестандартное форматирование и формулы в строках 1–3 допустимы, если:
- лист «Товары и цены» существует;
- нужные колонки присутствуют на правильных буквенных позициях;
- книга успешно читается и сохраняется без разрушения структуры;
- итоговый файл остаётся пригодным для загрузки в Ozon.

13.4.3. Формулы Ozon

Формулы в книге Ozon должны сохраняться без пересчёта Excel.
Принудительный пересчёт книги в Excel-совместимом движке не требуется.

13.5. Контракт Wildberries

Для Wildberries требуется сохранять:
- все колонки без изменений, кроме «Новая скидка»;
- порядок строк;
- формулы, если они есть;
- стили и форматирование, если технически поддерживаются используемой библиотекой;
- ширины колонок, фильтры, скрытые строки/колонки и объединения — по принципу максимального сохранения структуры книги.

Проектировщик и разработчик обязаны выбрать реализацию, которая не ломает структуру рабочего WB-файла.
Минимальный обязательный критерий приёмки:
- изменяется только колонка «Новая скидка»;
- остальные данные и структура файла пригодны для дальнейшего использования без ручного восстановления.

13.6. Проблемная колонка «Новая скидка» в Wildberries

Если колонка «Новая скидка»:
- защищена от записи;
- содержит формулы;
- имеет нестандартный формат, не позволяющий гарантированно записать итог,

то операция должна завершаться validation_failed/check_failed или failed с явной причиной невозможности безопасной записи.

14. WILDBERRIES

14.1. Назначение модуля

Модуль Wildberries рассчитывает итоговую скидку и записывает её в колонку «Новая скидка».

14.2. Входные файлы

Один запуск WB принимает:
- ровно 1 файл цен;
- от 1 до 20 файлов акций.

14.3. Лист

Если в книге один лист — используется он.
Если листов несколько — используется первый лист.

14.4. Обязательные колонки файла цен

- Артикул WB
- Текущая цена
- Новая скидка

14.5. Обязательные колонки файла акций

- Артикул WB
- Плановая цена для акции
- Загружаемая скидка для участия в акции

14.6. Нормализация

14.6.1. Артикул

- привести к строке;
- trim;
- удалить суффикс .0;
- пустое значение считать отсутствующим.

14.6.2. Числа

- удалить пробелы и неразрывные пробелы;
- заменить запятую на точку;
- нечисловое значение считать отсутствующим.

14.7. Агрегация по акциям

Для каждого артикула:
- min_discount = MIN(Загружаемая скидка)
- max_plan_price = MAX(Плановая цена для акции)

14.8. Формула

calculated_discount = ceil((1 - max_plan_price / current_price) * 100)

14.9. Арифметика

Для финального расчёта запрещено использовать float.
Необходимо использовать точную десятичную арифметику.

14.10. Гибридная логика

- final_discount_pre_threshold = MIN(min_discount, calculated_discount)

Если final_discount_pre_threshold > threshold_percent:
- final_discount = fallback_over_threshold_percent

Если артикул отсутствует во всех валидных файлах акций:
- final_discount = fallback_no_promo_percent

Иначе:
- final_discount = final_discount_pre_threshold

14.11. Ограничения результата

- итог всегда целое число;
- диапазон: 0–100 включительно.

14.12. Разрешённые изменения

Только колонка:
- Новая скидка

14.13. Критические ошибки запуска

- не загружен файл цен;
- не загружен ни один файл акций;
- файл не открывается как .xlsx;
- книга повреждена;
- нет обязательной колонки файла цен;
- все файлы акций невалидны;
- дубли в файле цен по Артикул WB;
- невозможность безопасной записи в колонку «Новая скидка».

14.14. Предупреждения по строкам

Файл цен:
- пустой Артикул WB;
- нет текущей цены;
- текущая цена <= 0;
- итог вне диапазона;
- иные некорректные данные строки.

Файлы акций:
- пустой Артикул WB;
- нет плановой цены;
- нет загружаемой скидки;
- нечисловые значения.

14.15. Сводный аудит WB

Обязательные показатели:
- всего строк;
- успешно обработано;
- предупреждения;
- пропущено строк;
- найдено в акциях;
- не найдено в акциях;
- применён fallback_no_promo;
- применён fallback_over_threshold;
- итог взят из min_discount;
- итог взят из calculated_discount;
- невалидные строки акций;
- невалидные файлы акций.

14.16. Краткий результат WB

Краткий результат включает:
- обработано строк;
- найдено в акциях;
- не найдено в акциях;
- предупреждения;
- при ошибке — короткую причину.

15. OZON

15.1. Назначение модуля

Модуль Ozon обрабатывает один Excel-файл акции формата «Эластичный бустинг» и меняет только допустимые колонки.

15.2. Вход

Ровно 1 файл .xlsx.

15.3. Лист

Обязательный лист:
- Товары и цены

15.4. Старт обработки

Строка 4.

15.5. Обязательные колонки по буквам

- J: Минимальная цена, RUB
- K: Участие товара в акции
- L: Итоговая цена по акции, RUB
- O: Цена для минимального акционного бустинга, RUB
- P: Цена для максимального акционного бустинга, RUB
- R: Остаток на складе Ozon, шт

15.6. Разрешённые изменения

Только:
- K
- L

15.7. Нормализация

Для J, O, P, R:
- общая нормализация чисел;
- * считается отсутствующим значением;
- пустая строка и None считаются отсутствующим значением.

15.8. Логика

Если J отсутствует:
- K = пусто
- L = пусто
- reason = missing_min_price

Если R отсутствует или <= 0:
- K = пусто
- L = пусто
- reason = no_stock

Если O и P одновременно отсутствуют:
- K = пусто
- L = пусто
- reason = no_boost_prices

Если P присутствует и P >= J:
- K = Да
- L = P
- reason = use_max_boost_price

Если P присутствует, O присутствует, P < J, O >= J:
- K = Да
- L = J
- reason = use_min_price

Если O присутствует и O < J:
- K = пусто
- L = пусто
- reason = below_min_price_threshold

Иначе:
- K = пусто
- L = пусто
- reason = insufficient_ozon_input_data

15.9. Требования к записи

- «Да» записывается строго как строка Да;
- не участвует записывается пустой ячейкой;
- числовые значения цены записываются как числа;
- целые числа могут записываться без дробной части.

15.10. Критические ошибки

- файл не загружен;
- загружено более одного файла;
- книга повреждена;
- нет листа «Товары и цены»;
- отсутствуют обязательные колонки по буквам;
- невозможность сохранить выходной файл.

15.11. Предупреждения

- недостаточно данных;
- отсутствует минимальная цена;
- отсутствует остаток;
- отсутствуют бусты;
- значения не распознаны как числа;
- исключение из акции по пороговой логике.

15.12. Сводный аудит Ozon

Обязательные показатели:
- всего строк данных;
- успешно обработано;
- предупреждения;
- участвуют;
- не участвуют;
- нет минимальной цены;
- нет остатка;
- нет бустов;
- use_max_boost_price;
- use_min_price;
- below_min_price_threshold;
- изменено строк.

15.13. Краткий результат Ozon

Краткий результат включает:
- изменено строк;
- участвуют;
- не участвуют;
- предупреждения;
- при ошибке — короткую причину.

16.2. Карточка запуска и страница детального аудита

Отдельная «карточка запуска» как второй самостоятельный экран не требуется.

16.3. Страница обработки

Должна позволять:
- выбрать магазин;
- загрузить файлы;
- видеть текущий активный набор файлов;
- удалить/заменить файл до запуска;
- запустить check;
- запустить process;
- видеть сводный аудит;
- видеть краткий предварительный просмотр детального аудита;
- перейти на полную страницу аудита;
- скачать результат после успешного process.

16.4. Устаревшая проверка перед process

Если текущая проверка устарела, отдельного ручного подтверждения перед process не требуется.
Система должна:
- автоматически выполнить новую валидацию;
- при необходимости показать, что прежний check недействителен;
- при неуспешной валидации завершить process как validation_failed.

16.5. История запусков

По умолчанию:
- сортировка от новых к старым;
- размер страницы 50;
- фильтры пустые.

18. БЕЗОПАСНОСТЬ

18.1. Авторизация обязательна.

18.2. Пароли хранятся только как безопасные хэши.

18.3. Прямой доступ к файлам по открытому URL запрещён.

18.4. Скачивание файла разрешено только пользователю, имеющему право на соответствующий магазин и запуск.

18.5. Первый Администратор создаётся через CLI-команду инициализации системы.
Через веб-интерфейс сценарий создания первого администратора не требуется.

18.6. Роли и базовые permissions создаются автоматически при инициализации системы.

18.7. Пользователь может изменить собственный пароль в рамках авторизованной сессии.

18.8. Принудительная смена пароля, восстановление доступа, сброс по email и self-service recovery в MVP не входят.

19.1. Целевое число одновременно работающих пользователей

- открытие истории: до 3 секунд
- открытие логов: до 3 секунд

- один файл акций Wildberries: до 25 МБ
- файл Ozon: до 25 МБ

20. КРИТЕРИИ ПРИЁМКИ

20.1. По данным и архитектуре

- реализован полный обязательный перечень сущностей;
- реализована модель roles + permissions + user_store_access;
- реализована модель runs с полями operation_type, lifecycle_status, business_result;
- реализовано асинхронное выполнение check/process;
- реализован polling статуса;
- реализованы блокировки по store_id + module_code.

20.2. По Wildberries

- корректно принимается набор файлов;
- корректно выполняется check;
- корректно выполняется process;
- правильно считаются скидки;
- запрещено использование float в финальном расчёте;
- меняется только колонка «Новая скидка»;
- формируются краткий результат, сводный и детальный аудит.

20.3. По Ozon

- корректно принимается файл;
- корректно выполняется check;
- корректно выполняется process;
- изменяются только K и L;
- структура книги не ломается;
- результат пригоден для обратной загрузки;
- формируются краткий результат, сводный и детальный аудит.

20.4. По интерфейсу

- роли, permissions и доступы соблюдаются;
- скрытие недоступных разделов соблюдается;
- история, логи и детальный аудит поддерживают серверные поиск, фильтры, сортировки и пагинацию;
- карточка запуска и полная страница аудита совпадают;
- пользователь без магазинов видит корректное пустое состояние.

20.5. По файлам

- временные файлы очищаются по правилам;
- исходные и результирующие файлы хранятся по правилам;
- старый успешный результат удаляется при появлении нового;
- факт удаления логируется;
- недоступность файлов корректно отражается в UI.
