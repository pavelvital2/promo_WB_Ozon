# 01_PROJECT_OVERVIEW_AND_AGENT_DOC_MAP.md

## 1. Краткий итог

Проект представляет собой веб-программу расчёта скидок для Wildberries и Ozon, работающую только через веб-интерфейс, без API маркетплейсов, без фоновых расписаний и без внешних интеграций в MVP.

Целевая архитектура должна обеспечивать:

- строгое разделение модулей Wildberries и Ozon;
- строгое разделение операций check и process;
- асинхронное выполнение операций через создание run и polling;
- полную трассируемость входных файлов, результата, сводного аудита, детального аудита и системных логов;
- модель доступа через roles + permissions + user_store_access;
- серверные поиск, фильтрацию, сортировку и пагинацию для history, logs и run detail audit;
- хранение детального аудита в PostgreSQL;
- файловое хранение во внутреннем каталоге VPS без прямого публичного URL;
- совместимость структуры документации с мультиагентной системой Codex CLI.

Система проектируется как монолитное веб-приложение с чётким внутренним модульным разделением. Внутри одного backend-контейнера логики выделяются отдельные прикладные контуры:
- auth / users / access control;
- stores;
- temporary uploads;
- runs orchestration;
- wb engine;
- ozon engine;
- audit;
- file storage;
- history;
- logs;
- admin CLI bootstrap.

Такой подход соответствует ТЗ, не расширяет scope MVP и остаётся пригодным для дальнейшей декомпозиции на задачи для нескольких агентов.

## 2. Цель архитектурной документации

Документация проекта должна служить одновременно для:
- оркестрации работ;
- проектирования;

- тестирования;

Каждый документ должен быть:
- автономным в своей предметной зоне;
- непротиворечивым относительно ТЗ;
- пригодным для чтения отдельным агентом без обязательной загрузки всей документации проекта;
- пригодным для последовательной декомпозиции в задачи.

## 3. Правило использования документации мультиагентной системой

Все агенты обязаны считать техническое задание единственным нормативным источником по бизнес-логике.

Архитектурная документация:
- не заменяет ТЗ;
- не изменяет ТЗ;
- конкретизирует реализацию;
- фиксирует границы модулей, слоёв, данных, маршрутов, жизненных циклов, тестовых контуров и зависимостей.

Если в документации встречается противоречие ТЗ, приоритет всегда у ТЗ.
Если агент видит реальный пробел, он не додумывает бизнес-правило самостоятельно, а поднимает это как архитектурный вопрос.

Архитектурный комплект документации считается готовым, когда:
- покрыты все обязательные разделы из Промта;
- не осталось нераскрытых обязательных сущностей ТЗ;
- все агенты имеют понятный набор документов для работы;
