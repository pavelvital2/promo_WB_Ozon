# 17_OPEN_ARCHITECTURAL_QUESTIONS.md

## 1. Назначение документа

Документ фиксирует только те архитектурные вопросы, которые:
- действительно остаются незакрытыми по ТЗ;
- нельзя безопасно домыслить без изменения проектного контракта;
- влияют на реализацию, приёмку или эксплуатационное поведение системы.

Документ не используется для расширения scope MVP.
Документ не вводит новые бизнес-правила.
Документ нужен только для фиксации реальных пробелов.

---

## 2. Краткий итог

На текущий момент критических архитектурных вопросов, которые нельзя безопасно закрыть без внешнего решения, не осталось.

Ранее открытый вопрос о retention historical run files закрыт проектным решением:
- historical run input/output files хранятся 5 дней;
- temporary files хранятся 24 часа.

Незакрытые проблемы проекта теперь относятся не к архитектурным пробелам ТЗ, а к незавершённым обязательным implementation blocks.
Они отдельно зафиксированы в:
- `18_REMEDIATION_PLAN_AFTER_AUDIT_2026_04_07.md`

---

## 3. Закрытый вопрос №1. Точный срок хранения run_files

## 3.1. Как вопрос был закрыт

- Historical run input/output files:
  - 5 дней
- Temporary files:
  - 24 часа

## 3.2. Что это теперь означает

Теперь можно однозначно определять:

1. `run_files.expires_at_utc`;
2. момент перехода в `expired`;
3. retention tests;
4. cleanup behavior;
5. UX для file availability.

---

## 4. Что не считается критическим открытым вопросом

Ниже перечислено, что **не** нужно выносить как отдельный критический пробел, потому что архитектура уже может это закрыть без изменения ТЗ.

## 4.1. Точные URL-маршруты

Не критично:
- можно зафиксировать проектно, сохраняя обязательный набор страниц и действий.

## 4.2. Внутренний формат JSON в summary/detail payload

Не критично:
- ТЗ требует состав данных, а не буквальный JSON-shape.

## 4.3. Точный выбор библиотек Python/Excel

Не критично:
- это техническое решение реализации, пока соблюдены workbook contracts.

## 4.4. Точный формат public_run_number

Не критично на уровне ТЗ:
- нужен уникальный публичный run identifier, конкретный шаблон может быть выбран проектно, если он стабилен и пригоден для UI/history/logs.

## 4.5. Точные page query param names

Не критично:
- важны server-side search/filter/sort/page semantics, а не конкретные имена параметров.

## 4.6. Дополнительные технические system log payload fields

Не критично:
- при условии, что обязательные event types и поля соблюдены.

## 5. Итоговая рекомендация по использованию этого документа

Документ больше не фиксирует активные архитектурные вопросы.

Использование:
- считать архитектурный retention-вопрос закрытым;
- не использовать этот файл как justification для откладывания retention implementation;
- дальнейшие незакрытые работы брать из remediation-плана:
  - `18_REMEDIATION_PLAN_AFTER_AUDIT_2026_04_07.md`

## 6. Граница документа

На текущий момент активных критических архитектурных вопросов не зафиксировано.

Если в будущем появится новый вопрос, который:
- нельзя безопасно закрыть проектным решением;
- влияет на ТЗ, приёмку или обязательный контракт,

он должен быть добавлен в этот документ как новый отдельный раздел.
