# 10_WB_PROCESSING_FLOW.md

## 1. Назначение документа

Документ фиксирует:
- полный прикладной поток модуля Wildberries;
- входной контракт файлов;
- правила check и process;
- этапы чтения, нормализации, агрегации и расчёта;
- правила записи результата в workbook;
- обязательные summary/detail audit данные;
- критические ошибки и предупреждения;
- инварианты, которые нельзя нарушать при реализации.

Документ является модульным архитектурным описанием WB-контура и не заменяет ТЗ.

---

## 2. Краткий итог

Модуль Wildberries принимает на один запуск:
- ровно 1 файл цен;
- от 1 до 20 файлов акций.

Модуль должен:
- прочитать книгу цен;
- прочитать все валидные книги акций;
- нормализовать данные;
- агрегировать акции по Артикулу WB;
- вычислить:
  - min_discount
  - max_plan_price
  - calculated_discount
  - final_discount_pre_threshold
  - final_discount
- сформировать summary audit;
- сформировать detail audit.

При `check`:
- расчёт выполняется полностью как модель результата;
- output file не создаётся;
- в workbook ничего не записывается.

При `process`:
- перед записью выполняется обязательная auto-validation;
- в output workbook изменяется только колонка `Новая скидка`;
- остальные данные и структура должны остаться пригодными для дальнейшего использования без ручного восстановления.

Ключевой инвариант:
- в финальном расчёте запрещено использовать float;
- должна использоваться точная десятичная арифметика.

---

## 3. Назначение WB-модуля

WB-модуль предназначен для расчёта итоговой скидки и записи её в колонку:
- `Новая скидка`

Модуль не должен:
- менять другие колонки;
- менять логику расчёта вне правил ТЗ;
- вводить альтернативные fallback-сценарии;
- использовать сторонние данные или API маркетплейса;
- выполнять действия вне загруженных пользователем файлов.

---

## 4. Входной контракт WB

## 4.1. Состав входа

Один WB run принимает:
- ровно 1 файл цен;
- от 1 до 20 файлов акций.

Это правило относится и к check, и к process.

## 4.2. Тип файлов

Поддерживаются только:
- валидные `.xlsx` файлы

Не поддерживаются:
- другие форматы;
- файлы с макросами, если приложение не может гарантировать корректное сохранение;
- защищённые книги/листы, если нельзя безопасно прочитать/записать требуемые данные;
- файлы, чья структура не позволяет гарантировать безопасную работу.

## 4.3. Ограничения размера

- максимум 25 МБ на один файл
- максимум 100 МБ суммарно на один WB run
- максимум 20 promo files

Нарушение этих ограничений должно блокировать запуск как file limit scenario.

---

## 5. Контракт workbook для WB

## 5.1. Выбор листа

Если в книге один лист:
- используется он.

Если листов несколько:
- используется первый лист.

Это правило относится:
- к файлу цен;
- к каждому файлу акций.

## 5.2. Обязательные колонки файла цен

Файл цен обязан содержать:
- `Артикул WB`
- `Текущая цена`
- `Новая скидка`

## 5.3. Обязательные колонки файла акций

Каждый файл акций обязан содержать:
- `Артикул WB`
- `Плановая цена для акции`
- `Загружаемая скидка для участия в акции`

## 5.4. Критерий пригодности workbook

Workbook пригоден для обработки только если:
- файл открывается как корректный `.xlsx`;
- книга не повреждена;
- обязательные колонки присутствуют;
- для process возможна безопасная запись в `Новая скидка`.

---

## 6. Общий pipeline WB

## 6.1. Check pipeline

WB check состоит из следующих стадий:

1. Проверка доступности и состава файлов
2. Открытие файла цен
3. Проверка структуры файла цен
4. Открытие файлов акций
5. Проверка структуры файлов акций
6. Нормализация строк файла цен
7. Нормализация строк файлов акций
8. Выявление невалидных строк
9. Агрегация валидных строк акций по Артикулу WB
10. Расчёт прогнозного результата по каждой строке файла цен
11. Формирование summary audit
12. Формирование detail audit
13. Формирование short_result_text
14. Завершение run без output file

## 6.2. Process pipeline

WB process состоит из следующих стадий:

1. Auto-validation
2. Открытие файла цен
3. Проверка возможности безопасной записи в `Новая скидка`
4. Открытие и валидация файлов акций
5. Нормализация данных
6. Агрегация по акциям
7. Расчёт результата по строкам файла цен
8. Запись только колонки `Новая скидка`
9. Сохранение output workbook
10. Формирование summary audit
11. Формирование detail audit
12. Формирование short_result_text
13. Завершение successful/failed process

---

## 7. Граница check для WB

## 7.1. Что check обязан делать

На этапе check для Wildberries разрешено и требуется:
- проверять структуру файла цен;
- проверять структуру файлов акций;
- нормализовать данные;
- выявлять невалидные строки;
- определять наличие артикула в акциях;
- считать min_discount;
- считать max_plan_price;
- считать calculated_discount;
- считать итоговое значение final_discount_pre_threshold;
- определять, превысился бы threshold;
- определять, применился бы fallback_no_promo;
- определять, применился бы fallback_over_threshold;
- формировать краткий результат, сводный аудит и детальный аудит так, как если бы расчёт был выполнен.

## 7.2. Что check запрещено делать

Check не должен:
- записывать изменения в result workbook;
- формировать output file;
- менять исходный файл;
- обновлять `Новая скидка`.

---

## 8. Граница process для WB

## 8.1. Что process обязан делать

Process обязан:
- выполнить обязательную auto-validation;
- выполнить полный расчёт по правилам WB;
- записать результат в колонку `Новая скидка`;
- сохранить output workbook;
- сформировать summary audit;
- сформировать detail audit;
- сформировать short_result_text.

## 8.2. Что process запрещено делать

Process не должен:
- менять какие-либо колонки, кроме `Новая скидка`;
- менять порядок строк;
- разрушать формулы;
- разрушать пригодность workbook для дальнейшего использования;
- использовать float в финальном расчёте.

---

## 9. Нормализация WB данных

## 9.1. Нормализация артикула

Для поля `Артикул WB` применяется следующая нормализация:
- привести к строке;
- trim;
- удалить суффикс `.0`;
- пустое значение считать отсутствующим.

## 9.2. Нормализация чисел

Для числовых полей применяется:
- удалить обычные пробелы;
- удалить неразрывные пробелы;
- заменить запятую на точку;
- нечисловое значение считать отсутствующим.

## 9.3. Какие поля считаются числовыми

В WB-модуле числовая нормализация применяется как минимум к:
- `Текущая цена`
- `Плановая цена для акции`
- `Загружаемая скидка для участия в акции`

## 9.4. Архитектурный принцип

Нормализация должна выполняться одинаково в:
- check
- process

Чтобы:
- check и process давали согласованный результат;
- warning/error semantics не расходились между режимами.

---

## 10. Валидность строк файла цен

## 10.1. Для чего используются строки файла цен

Строки файла цен являются основными строками расчёта.
Итоговая `Новая скидка` определяется именно по строкам файла цен.

## 10.2. Предупреждения по строкам файла цен

Для строки файла цен должны выявляться предупреждения при следующих ситуациях:
- пустой Артикул WB;
- нет текущей цены;
- текущая цена <= 0;
- итог вне диапазона;
- иные некорректные данные строки.

## 10.3. Поведение для проблемной строки

Если строка файла цен не позволяет корректно рассчитать итог:
- строка должна быть отражена в detail audit;
- run в целом не обязательно должен завершиться failed, если проблема локальна и не является глобальной критической ошибкой;
- итоговая классификация зависит от общего числа проблем и финального summary.

---

## 11. Валидность строк файлов акций

## 11.1. Назначение строк акций

Строки акций используются как источник для агрегирования по артикулу:
- min_discount
- max_plan_price

## 11.2. Предупреждения по строкам акций

Для строк акций должны выявляться warning-level проблемы:
- пустой Артикул WB;
- нет плановой цены;
- нет загружаемой скидки;
- нечисловые значения.

## 11.3. Влияние невалидной строки акции

Невалидная строка акции:
- не должна участвовать в агрегации;
- должна быть отражена в detail audit;
- должна учитываться в summary counters:
  - невалидные строки акций

## 11.4. Влияние невалидного promo file

Если promo file не проходит структурную валидацию:
- он считается невалидным файлом акций;
- должен учитываться в summary counter:
  - невалидные файлы акций

---

## 12. Агрегация по акциям

## 12.1. Единица агрегации

Агрегация выполняется по:
- нормализованному `Артикул WB`

## 12.2. Вычисляемые агрегаты

Для каждого артикула:
- `min_discount = MIN(Загружаемая скидка)`
- `max_plan_price = MAX(Плановая цена для акции)`

## 12.3. Источник агрегатов

В агрегации участвуют только:
- валидные строки
- валидных promo files

## 12.4. Отсутствие артикула в акциях

Если артикул отсутствует во всех валидных файлах акций:
- это отдельный доменный сценарий;
- применяется fallback_no_promo_percent.

---

## 13. Формула WB

## 13.1. Обязательная формула

`calculated_discount = ceil((1 - max_plan_price / current_price) * 100)`

## 13.2. Требование к арифметике

Для финального расчёта запрещено использовать float.

Обязательно:
- точная десятичная арифметика

Следовательно:
- деление, умножение и округление вверх должны выполняться в decimal-safe режиме.

## 13.3. Требование к округлению

Используется:
- `ceil`

Никакое другое округление:
- floor
- bankers rounding
- standard round  
не допускается.

---

## 14. Гибридная логика WB

## 14.1. Первый этап

Сначала вычисляется:
- `final_discount_pre_threshold = MIN(min_discount, calculated_discount)`

## 14.2. Проверка threshold

Если:
- `final_discount_pre_threshold > threshold_percent`

то:
- `final_discount = fallback_over_threshold_percent`

## 14.3. Сценарий отсутствия артикула в акциях

Если артикул отсутствует во всех валидных файлах акций:
- `final_discount = fallback_no_promo_percent`

## 14.4. Обычный сценарий

Иначе:
- `final_discount = final_discount_pre_threshold`

## 14.5. Приоритет трактовки

Архитектура реализации должна строго сохранить последовательность условий из ТЗ.
Нельзя:
- переставлять условия местами;
- сначала применять threshold, если артикула вообще нет в акциях;
- вводить дополнительные промежуточные ветки.

---

## 15. Ограничения результата WB

## 15.1. Тип результата

Итог:
- всегда целое число

## 15.2. Диапазон результата

Итоговый результат должен находиться в диапазоне:
- от 0 до 100 включительно

## 15.3. Сценарий выхода за диапазон

Если итог вне диапазона:
- это должно быть отражено в warning/error semantics по строке;
- строка должна попасть в detail audit;
- run outcome определяется общим контекстом обработки.

---

## 16. Использование store settings в WB

## 16.1. Обязательные store settings для WB

Для WB store обязательны:
- `wb_threshold_percent`
- `wb_fallback_no_promo_percent`
- `wb_fallback_over_threshold_percent`

## 16.2. Источник этих значений

WB-модуль должен брать эти значения:
- из store settings соответствующего магазина

## 16.3. Что запрещено

Запрещено:
- использовать захардкоженные значения вместо store settings;
- брать настройки из временных файлов;
- менять store settings внутри run.

---

## 17. Критические ошибки WB

## 17.1. Закрытый перечень обязательных критических ошибок запуска

WB run должен завершаться failed/check_failed/validation_failed/failed при сценариях:

- не загружен файл цен;
- не загружен ни один файл акций;
- файл не открывается как `.xlsx`;
- книга повреждена;
- нет обязательной колонки файла цен;
- все файлы акций невалидны;
- дубли в файле цен по Артикул WB;
- невозможность безопасной записи в колонку `Новая скидка`.

## 17.2. Дубли в файле цен

Дубли по `Артикул WB` в файле цен являются критической ошибкой запуска.

Следствие:
- check должен завершиться `check_failed`
- process должен завершиться `validation_failed` либо `failed` в зависимости от стадии, но в каноническом process path это должно быть остановлено на validating

## 17.3. Все promo files невалидны

Если все файлы акций невалидны:
- run не может продолжить расчёт как обычный сценарий;
- это критическая ошибка запуска.

---

## 18. Невозможность безопасной записи в `Новая скидка`

## 18.1. Когда сценарий считается критическим

Если колонка `Новая скидка`:
- защищена от записи;
- содержит формулы;
- имеет нестандартный формат, не позволяющий гарантированно записать итог,

то операция должна завершаться ошибкой невозможности безопасной записи.

## 18.2. Для каких режимов это обязательно

### Для check
Такой сценарий должен приводить к:
- `check_failed`

### Для process
Такой сценарий должен приводить к:
- `validation_failed` или `failed` в зависимости от стадии execution,
но архитектурно предпочтительно отлавливать это в validating до реальной записи.

## 18.3. Принцип реализации

Если система не может гарантировать безопасную запись:
- она обязана отказаться от записи,
а не “попробовать и сохранить как получится”.

---

## 19. Правила работы с workbook структуры WB

## 19.1. Что необходимо сохранить

Для Wildberries требуется сохранять:
- все колонки без изменений, кроме `Новая скидка`;
- порядок строк;
- формулы, если они есть;
- стили и форматирование, если технически поддерживаются используемой библиотекой;
- ширины колонок, фильтры, скрытые строки/колонки и объединения — по принципу максимального сохранения структуры книги.

## 19.2. Минимальный обязательный критерий приёмки

Минимальный обязательный критерий:
- изменяется только колонка `Новая скидка`;
- остальные данные и структура файла пригодны для дальнейшего использования без ручного восстановления.

## 19.3. Следствие для реализации

Выбранный Excel writer для WB обязан быть подобран под safe write сценарий.
Нельзя выбирать реализацию, которая:
- переформатирует workbook целиком;
- ломает рабочий файл;
- меняет скрытые элементы/фильтры/порядок данных сверх допустимого.

---

## 20. WB check summary audit

## 20.1. Обязательные показатели

WB summary audit обязан включать:
- всего строк;
- успешно обработано;
- предупреждения;
- пропущено строк;
- найдено в акциях;
- не найдено в акциях;
- применён fallback_no_promo;
- применён fallback_over_threshold;
- итог взят из min_discount;
- итог взят из calculated_discount;
- невалидные строки акций;
- невалидные файлы акций.

## 20.2. Для check и process

Этот набор должен быть доступен и для:
- check
- process

Потому что check моделирует итог так, как если бы расчёт был выполнен.

## 20.3. Назначение summary

Summary должен позволять понять:
- сколько строк было реально пригодно к обработке;
- сколько строк нашли акции;
- где сработали fallback-ветки;
- насколько качественен входной набор.

---

## 21. WB short result

## 21.1. Обязательный состав

Краткий результат WB должен включать:
- обработано строк;
- найдено в акциях;
- не найдено в акциях;
- предупреждения;
- при ошибке — короткую причину.

## 21.2. Роль short result

Short result используется для:
- history list
- dashboard
- run header
- polling result after completion

Он не заменяет summary audit.

---

## 22. WB detail audit

## 22.1. Обязательные колонки UI

Для WB detail audit обязательны:
- row_number
- Артикул WB
- Текущая цена
- min_discount
- max_plan_price
- calculated_discount
- final_discount_pre_threshold
- final_discount
- severity
- decision_reason
- message

## 22.2. Источник данных для detail audit

Эти поля должны формироваться из:
- общих detail audit полей
- WB audit payload

UI не должен пересчитывать их из workbook.

## 22.3. Логика severity для detail rows

### `info`
Нормальная строка с вычисленным результатом и объяснением решения.

### `warning`
Строка обработана с отклонением или пропуском части данных:
- нет цены;
- нет артикула;
- не найдено в акциях;
- fallback scenario;
- итог вне диапазона;
- иные некорректные данные строки.

### `error`
Строка содержит ошибку уровня строки, делающую её непригодной к расчёту как нормальную строку.

Важно:
- наличие error rows не обязательно делает весь run failed;
- global outcome определяется общим контекстом и критическими ошибками запуска.

## 23. Рекомендуемый набор WB decision_reason

- result_out_of_range

## 24. Check outcome mapping для WB

## 24.1. `check_passed`

Используется, если:
- критических ошибок запуска нет;
- warning-level отклонения отсутствуют или не зафиксированы как предупреждения итогового запуска.

## 24.2. `check_passed_with_warnings`

Используется, если:
- расчётный check выполнен;
- есть warnings по строкам/акциям/частичным данным;
- запуск остаётся пригодным как проверка.

## 24.3. `check_failed`

Используется, если:
- возникла критическая ошибка запуска;
- невозможно корректно выполнить WB check.

## 25. Process outcome mapping для WB

## 25.1. `validation_failed`

Используется, если на auto-validation стадии выявлено:
- отсутствие обязательных файлов;
- невалидный workbook;
- дубли в файле цен;
- отсутствие обязательных колонок;
- все promo files невалидны;
- невозможность безопасной записи в `Новая скидка`.

## 25.2. `completed`

Используется, если:
- process завершён успешно;
- output file сформирован;
- существенных warning-level итогов нет.

## 25.3. `completed_with_warnings`

Используется, если:
- process завершён успешно;
- output file сформирован;
- есть warning-level отклонения по строкам/данным.

## 25.4. `failed`

Используется, если:
- execution сорвался по системной или прикладной причине после стадии принятия и/или processing stage.

## 26. Связь WB detail audit с summary counters

WB summary counters должны агрегироваться из того же доменного результата, который формирует detail audit.

Нельзя допускать расхождение, когда:
- detail rows показывают одно;
- summary audit считает другое;
- short result сообщает третье.

Следовательно:
- summary builder и detail builder должны опираться на единый canonical processing result.

## 27. Пример канонической последовательности обработки строки файла цен

Для каждой строки файла цен канонический порядок должен быть таким:

1. Прочитать row_number
2. Нормализовать Артикул WB
3. Нормализовать Текущую цену
4. Проверить валидность строки цены
5. Найти агрегат по Артикулу WB среди валидных promo data
6. Если не найдено:
   - применить fallback_no_promo
7. Если найдено:
   - вычислить calculated_discount
   - вычислить final_discount_pre_threshold
   - проверить threshold
   - либо применить fallback_over_threshold
   - либо взять final_discount_pre_threshold
8. Проверить диапазон результата
9. Записать detail audit row
10. Учесть строку в summary counters
11. При process — записать final_discount в `Новая скидка`

Изменять этот порядок без необходимости нельзя, потому что это может поменять трактовку результата.
