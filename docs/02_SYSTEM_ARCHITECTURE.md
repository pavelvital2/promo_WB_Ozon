# 02_SYSTEM_ARCHITECTURE.md

## 1. Назначение документа

Документ фиксирует целевую архитектуру системы:
- границы решения;
- основные подсистемы;
- внутренние контуры;
- взаимодействие между слоями;
- модель выполнения check/process;
- место WB и Ozon в общей структуре;
- принципы трассируемости, аудита, логирования и файлового хранения.

Документ не заменяет ТЗ и не детализирует:
- полную схему БД;
- детальную матрицу доступа;
- полные маршруты UI;
- этапы и задачи реализации.

Эти аспекты раскрываются в последующих документах.

---

## 2. Краткий итог

Целевая система проектируется как серверное веб-приложение на Python, работающее на Ubuntu VPS, использующее PostgreSQL, nginx и systemd, без Docker и без внешних интеграций в MVP.

Архитектурный тип:
- монолитное веб-приложение;
- с жёстким внутренним модульным разделением;
- с асинхронным исполнением операций через run + polling;
- с выделенными прикладными контурами для:
  - авторизации и пользователей;
  - магазинов и доступа;
  - временных файлов;
  - run orchestration;
  - WB-обработки;
  - Ozon-обработки;
  - аудита;
  - логов;
  - истории;
  - файлового хранения;
  - административной инициализации.

Выбранная архитектура соответствует ТЗ, потому что:
- не вводит внешние сервисы и интеграции;
- не расширяет MVP;
- сохраняет разделение WB и Ozon;
- сохраняет разделение check и process;
- поддерживает асинхронную обработку;
- обеспечивает воспроизводимость run;
- обеспечивает серверную работу с историей, логами и детальным аудитом;
- позволяет мультиагентной системе реализовывать проект по модульным границам без конфликтов.

## 3. Архитектурные принципы

## 3.1. Принцип единственного нормативного источника

Бизнес-логика определяется только ТЗ.
Архитектура:
- конкретизирует реализацию;
- не меняет правила расчёта;
- не добавляет альтернативные сценарии;
- не расширяет перечни сущностей, статусов, результатов и UI-сценариев.

## 3.2. Принцип модульной изоляции

Внутри приложения должны быть логически отделены:
- модуль Wildberries;
- модуль Ozon;
- access control;
- file storage;
- audit;
- logs;
- history.

Это необходимо, чтобы:
- не смешивать Excel-контракты WB и Ozon;
- не смешивать check/process логику разных модулей;
- не допустить несанкционированного изменения общего поведения при локальной доработке.

## 3.3. Принцип воспроизводимости run

Каждый run должен быть воспроизводим как исторический объект:
- известен магазин;
- известен пользователь-инициатор;
- известен operation_type;
- известен lifecycle_status;
- известен business_result;
- известен exact input set;
- известны все входные файлы этого run;
- известен итоговый результат;
- известен сводный аудит;
- известен детальный аудит;
- известны системные события, связанные с run.

## 3.4. Принцип изоляции временного набора файлов

До запуска check/process система работает только с активным временным набором файлов в контуре:
- пользователь;
- магазин;
- модуль.

Этот контур отделён от исторических run-файлов.
Временный набор:
- изменяем;
- очищаем по retention;
- не является run сам по себе;
- становится историческим только после фактического запуска операции.

## 3.5. Принцип асинхронного исполнения

HTTP-запрос не выполняет всю операцию целиком.
HTTP-слой только:
- валидирует доступ и базовые ограничения;
- создаёт run;
- ставит задачу на выполнение;
- возвращает ответ о принятии в обработку.

Фактическое выполнение идёт отдельно, а UI получает результат через polling.

## 3.6. Принцип минимально допустимых внутренних технических компонентов

Допускаются только такие технические внутренние компоненты, которые:
- не создают новых бизнес-сущностей MVP;
- не меняют пользовательский контракт;
- служат реализации обязательных требований ТЗ.

## 4. Верхнеуровневая схема системы

Система состоит из следующих верхних уровней:

1. Веб-уровень
   - страницы;
   - формы;
   - polling;
   - серверный рендеринг или эквивалентный серверный HTTP UI;
   - обработка пользовательских сценариев.

2. HTTP / application entry layer
   - маршруты;
   - авторизация;
   - проверка доступа;
   - валидация входных команд;
   - создание команд в application layer.

3. Application layer
   - use cases;
   - orchestration сервисы;
   - координация жизненного цикла run;
   - выполнение бизнес-сценариев;
   - управление транзакционными границами.

4. Domain / business rules layer
   - чистая логика WB;
   - чистая логика Ozon;
   - нормализация данных;
   - расчёты;
   - формирование аудита;
   - формирование короткого результата.

5. Infrastructure layer
   - PostgreSQL repositories;
   - файловое хранилище;
   - Excel reader/writer;
   - асинхронный task runner;
   - системное журналирование;
   - CLI bootstrap;
   - password hashing;
   - clock/timezone conversion.

## 5. Верхнеуровневые подсистемы

## 5.1. Web UI subsystem

Назначение:
- отрисовка обязательных страниц;
- формы логина;
- работа с пользователями и магазинами;
- экраны обработки WB/Ozon;
- история запусков;
- логи;
- экран run/detail audit;
- polling run-статуса;
- скачивание файлов с серверной проверкой прав.

Важные ограничения:
- недоступные разделы скрываются полностью;
- история, логи и детальный аудит работают только через серверные фильтры/поиск/сортировку/пагинацию;
- пользователь без магазинов видит пустое состояние и не получает доступ к модулям запуска.

## 5.2. Identity and access subsystem

Назначение:
- аутентификация;
- проверка блокировки пользователя;
- загрузка роли;
- загрузка permissions;
- проверка user_store_access;
- enforcement доступа в каждом use case;
- скрытие разделов UI по модели доступа.

Состав:
- auth service;
- current user context;
- role/permission resolver;
- store access resolver;
- authorization policy checks.

## 5.3. Stores management subsystem

Назначение:
- создание магазинов;
- редактирование магазинов;
- архивирование/восстановление;
- хранение настроек WB для расчёта;
- контроль доступности магазина для новых запусков.

Особенность:
- store определяет marketplace и module_code;
- archived store остаётся в истории, но исключается из новых запусков.

## 5.4. Temporary uploads subsystem

Назначение:
- загрузка временных файлов;
- хранение активного набора;
- удаление/замена временного файла;
- расчёт сигнатуры набора;
- очистка просроченных временных файлов;
- журналирование операций над временными файлами.

Особенность:
- один активный набор на пару пользователь + магазин + модуль;
- временный набор не является частью history, пока не произошёл run.

## 5.5. Run orchestration subsystem

Назначение:
- создание run;
- выбор допустимого сценария check/process;
- захват блокировки по store_id + module_code;
- перевод lifecycle_status;
- запуск check/process worker execution;
- запись business_result;
- завершение run;
- связь run с input/output файлами, аудитом и логами.

Это центральный координирующий контур системы.

## 5.6. WB processing subsystem

Назначение:
- чтение и валидация одного файла цен и 1–20 файлов акций;
- нормализация данных;
- агрегация по акциям;
- check-расчёт без записи результата;
- process-расчёт с записью только в «Новая скидка»;
- формирование сводного и детального аудита;
- формирование краткого результата;
- защита от unsafe write в WB-файл.

## 5.7. Ozon processing subsystem

Назначение:
- чтение одного файла Ozon;
- проверка листа и колонок по буквенным позициям;
- нормализация входных значений;
- check-моделирование результата без сохранения файла;
- process-обработка с изменением только колонок K и L;
- формирование сводного и детального аудита;
- формирование краткого результата;
- сохранение структуры книги пригодной для обратной загрузки.

## 5.8. Audit subsystem

Назначение:
- единое хранение check-аудита и process-аудита;
- запись summary audit;
- запись detail audit;
- обеспечение серверного поиска/фильтрации/сортировки/пагинации;
- подготовка данных для страницы run/detail.

## 5.9. Logging subsystem

Назначение:
- запись обязательных системных событий;
- связывание событий с user/store/run/module;
- поддержка server-side поиска и фильтров;
- отделение системных логов от run-audit.

Важно:
- audit не заменяет system_logs;
- system_logs не заменяет audit.

## 5.10. File storage subsystem

Назначение:
- физическое хранение временных файлов;
- физическое хранение input copies per run;
- физическое хранение output files per run;
- контроль доступности файлов;
- управление retention;
- перевод файлов в unavailable;
- логирование superseded/deleted scenarios.

## 5.11. Admin bootstrap subsystem

Назначение:
- CLI-инициализация первого администратора;
- создание базовых ролей;
- создание базовых permissions;
- bootstrap состояния системы, обязательного по ТЗ.

## 6. Целевая логическая диаграмма взаимодействия

Ниже фиксируется канонический поток взаимодействия подсистем.

### 6.1. Общий пользовательский поток

1. Пользователь проходит авторизацию.
2. UI получает контекст доступа.
3. Пользователь попадает:
   - на dashboard;
   - затем в доступный раздел.
4. Для страницы обработки:
   - выбирается магазин;
   - загружаются временные файлы;
   - показывается текущий активный набор.
5. По команде check/process:
   - HTTP слой проверяет права;
   - проверяет статус магазина;
   - проверяет базовые файловые ограничения;
   - запрашивает orchestration use case.
6. Orchestration layer:
   - создаёт run;
   - фиксирует старт;
   - ставит задачу на асинхронное выполнение.
7. Worker execution:
   - берёт lock по store + module;
   - читает текущий набор;
   - копирует входные файлы в run/input;
   - выполняет check/process;
   - пишет audits;
   - при process пишет output;
   - завершает run.
8. UI по polling получает:
   - lifecycle_status;
   - business_result;
   - short_result_text.
9. После завершения пользователь:
   - смотрит сводный аудит;
   - открывает детальный аудит;
   - скачивает результат при успешном process;
   - при наличии доступа может скачать input files из history в пределах срока хранения.

## 7. Слои архитектуры и правила зависимостей

## 7.1. Presentation layer

Содержит:
- страницы;
- HTTP handlers/controllers;
- form parsers;
- response mappers;
- polling endpoints.

Не должен:
- напрямую читать Excel;
- напрямую писать в БД бизнес-объекты;
- напрямую реализовывать WB/Ozon-логику.

## 7.2. Application layer

Содержит:
- use cases;
- orchestration services;
- authorization checks на уровне сценария;
- coordination между repositories, storage и engines.

Не должен:
- включать низкоуровневый код работы с Excel;
- содержать SQL как основную бизнес-логику;
- хранить конкретные UI-представления.

## 7.3. Domain layer

Содержит:
- правила валидации доменных сценариев;
- правила check/process;
- нормализацию WB/Ozon;
- расчёты;
- доменные reason/decision модели;
- формирование domain audit payload.

Не должен:
- знать о HTTP;
- зависеть от UI;
- зависеть от конкретной БД.

## 7.4. Infrastructure layer

Содержит:
- PostgreSQL adapters;
- file system storage adapters;
- Excel workbook adapters;
- async execution adapters;
- password hasher;
- logging sink;
- CLI initializers.

Не должен:
- сам принимать бизнес-решения;
- подменять доменные правила.

## 9. Разделение WB и Ozon в архитектуре

Разделение WB и Ozon обязательно на следующих уровнях:

1. UI-страницы обработки
   - отдельная страница обработки Wildberries;
   - отдельная страница обработки Ozon.

2. Application use cases
   - отдельные use cases check/process для WB;
   - отдельные use cases check/process для Ozon.

3. Domain rules
   - отдельный движок расчёта WB;
   - отдельный движок обработки Ozon.

4. Excel adapters
   - отдельные reader/writer контракты;
   - отдельные правила safe write.

5. Audit composition
   - общий механизм хранения;
   - разные payload structures;
   - разные обязательные колонки в detail UI.

6. File role semantics
   - отдельные file_role значения;
   - отдельный input contract.

Разделение должно быть жёстким.
Общий код допускается только для:
- инфраструктуры;
- репозиториев;
- общих технических утилит;
- общих моделей run/audit/log/file access.

## 10. Разделение check и process в архитектуре

Check и process являются отдельными доменными сценариями, даже если используют общие части пайплайна.

Обязательное архитектурное правило:
- check и process не должны быть одной и той же операцией с флагом “dry_run”.

Допустима общая техническая переиспользуемая база, но прикладной слой обязан иметь:
- отдельный use case check;
- отдельный use case process;
- отдельные lifecycle transition rules;
- отдельные business_result sets.

Для process обязательна встроенная автоматическая валидация как внутренняя фаза того же run, а не как отдельный run.

## 11. Модель асинхронного выполнения

## 11.1. Общая схема

Асинхронная модель состоит из четырёх частей:

1. HTTP acceptance
   - принять запрос;
   - проверить доступ;
   - проверить магазин;
   - проверить ограничения;
   - создать run.

2. Task scheduling
   - передать run в backend execution.

3. Backend execution
   - взять lock;
   - выполнить check/process;
   - завершить run.

4. Polling
   - UI периодически запрашивает статус run;
   - интервал fixed: 2 секунды.

## 11.2. Внутренний исполнитель

ТЗ не требует внешней очереди.
Поэтому допустим внутренний backend execution component в составе приложения, если он:
- не меняет пользовательский контракт;
- поддерживает асинхронную модель;
- укладывается в ограничения MVP;
- обеспечивает контроль статусов и таймаутов.

## 11.3. Lock manager

Архитектурно обязателен внутренний компонент блокировки по паре:
- store_id;
- module_code.

Его задача:
- не допустить два активных run на одну пару;
- возвращать controlled conflict;
- гарантировать согласованность process output chain.

## 12. Модель хранения и трассируемости файлов

Архитектура файлов должна разделять три контура:

### 12.1. Temporary storage contour

Используется до запуска.
Каталог:
- `/storage/uploads/tmp/`

Содержит:
- только временные пользовательские загрузки;
- активные наборы;
- файлы с TTL 24 часа;
- файлы, не являющиеся частью history, пока не произошёл run.

### 12.2. Run input storage contour

Используется после старта run.
Каталог:
- `/storage/runs/{module_code}/{store_id}/{public_run_number}/input/`

Содержит:
- отдельные физические копии всех входных файлов run;
- неизменяемый исторический набор входов данного run.

### 12.3. Run output storage contour

Используется для process-результатов.
Каталог:
- `/storage/runs/{module_code}/{store_id}/{public_run_number}/output/`

Содержит:
- только выходные файлы run;
- в check output file не создаётся.

## 13. Модель аудита и журналирования в общей архитектуре

## 13.1. Run audit

Каждый run обязан породить:
- один summary audit;
- множество detail audit rows.

Audit нужен для:
- объяснимости результата;
- отображения на UI;
- исторической воспроизводимости;
- серверной работы с большим объёмом построчных записей.

## 13.2. System logs

System logs предназначены не для построчного результата обработки, а для событий системы:
- вход;
- создание/изменение сущностей;
- запуск/завершение операций;
- очистка временных файлов;
- superseded result handling;
- системные ошибки.

## 13.3. Архитектурное правило разделения

Нельзя смешивать:
- detail audit как лог,
- system_logs как результат построчной обработки.

Это разные сущности и разные UX-сценарии.

## 14. Модель UI в общей архитектуре

UI проектируется как thin server-driven interface.

UI не должен:
- дублировать бизнес-логику на клиенте;
- самостоятельно считать скидки;
- самостоятельно определять итоговые статусы run;
- загружать весь audit целиком.

UI должен:
- инициировать сценарии;
- показывать статусы;
- отображать доступные действия;
- делать polling;
- запрашивать серверные таблицы с фильтрами и пагинацией;
- корректно показывать недоступность файлов и пустые состояния.

## 15. Нефункциональная архитектурная база

Система проектируется под следующие ограничения:

- до 20 одновременно активных пользователей;
- до 200 000 строк детального аудита на один run;
- целевой быстрый отклик UI-страниц;
- server-side pagination для тяжёлых таблиц;
- отдельные индексы по search/filter/sort полям;
- ежедневный backup БД, файлов и логов;
- ротация логов минимум 30 дней.

Это означает, что архитектура обязана:
- избегать загрузки полного audit набора в память UI-ответа;
- исключать синхронное ожидание долгих задач в HTTP;
- иметь жёсткие границы между metadata и heavy data;
- обеспечивать выборку истории и логов по индексируемым полям.

## 16. Архитектурные инварианты

Ниже перечислены инварианты, которые считаются обязательными для всей системы:

1. Нет внешних API и внешних интеграций в MVP.
2. Нет прямого публичного URL-доступа к файлам.
3. Нет нескольких активных run по одной паре store + module.
4. Нет расширения списка run lifecycle_status и business_result.
5. Нет хранения детального аудита только в файлах.
6. Нет синхронной долгой обработки через HTTP вместо run + polling.
7. Нет смешения модулей WB и Ozon в одном processing engine.
8. Нет смешения check и process в один сценарий без разделения статусов и результатов.
9. Нет изменения неразрешённых колонок Excel-файлов.
10. Нет удаления исторических runs, audits и logs.
11. Нет выбора старых временных наборов как параллельных активных сценариев.
12. Нет UI-разделов, видимых без фактического права доступа.
