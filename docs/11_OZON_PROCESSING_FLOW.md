# 11_OZON_PROCESSING_FLOW.md

## 1. Назначение документа

Документ фиксирует:
- полный прикладной поток модуля Ozon;
- входной контракт файла;
- обязательный лист и обязательные колонки;
- правила check и process;
- этапы чтения, нормализации и принятия решения по строкам;
- правила записи результата в workbook;
- обязательные summary/detail audit данные;
- критические ошибки и предупреждения;
- инварианты, которые нельзя нарушать при реализации.

Документ является модульным архитектурным описанием Ozon-контура и не заменяет ТЗ.

---

## 2. Краткий итог

Модуль Ozon принимает на один запуск:
- ровно 1 файл `.xlsx`

Модуль должен:
- открыть книгу;
- найти обязательный лист `Товары и цены`;
- проверить обязательные колонки строго по буквенным позициям:
  - J
  - K
  - L
  - O
  - P
  - R
- читать данные начиная со строки 4;
- нормализовать значения;
- по каждой строке определить один из сценариев решения;
- сформировать summary audit;
- сформировать detail audit.

При `check`:
- логика обработки моделируется полностью;
- output file не создаётся;
- в workbook ничего не записывается.

При `process`:
- перед записью выполняется обязательная auto-validation;
- изменяются только колонки:
  - K
  - L
- структура книги не должна ломаться;
- итоговый файл должен оставаться пригодным для обратной загрузки в Ozon.

---

## 3. Назначение Ozon-модуля

Ozon-модуль обрабатывает один Excel-файл акции формата:
- `Эластичный бустинг`

Модуль должен менять только допустимые колонки:
- K
- L

Модуль не должен:
- изменять другие колонки;
- пересчитывать книгу принудительно в Excel-совместимом движке;
- менять логику принятия решения вне правил ТЗ;
- использовать сторонние API или внешние данные.

---

## 4. Входной контракт Ozon

## 4.1. Состав входа

Один Ozon run принимает:
- ровно 1 файл `.xlsx`

Это правило относится и к check, и к process.

## 4.2. Критическая ошибка по составу входа

Критическими ошибками являются:
- файл не загружен;
- загружено более одного файла.

## 4.3. Ограничение размера

Максимальный размер Ozon-файла:
- 25 МБ

Нарушение лимита должно блокировать запуск как file limit scenario.

---

## 5. Контракт workbook для Ozon

## 5.1. Обязательный лист

Книга обязана содержать лист:
- `Товары и цены`

Если такого листа нет:
- это критическая ошибка запуска.

## 5.2. Старт обработки

Обработка строк начинается:
- со строки 4

Строки 1–3 являются служебной частью шаблона и не должны считаться строками данных.

## 5.3. Обязательные колонки по буквенным позициям

Первичный критерий корректности шаблона — наличие обязательных колонок на правильных буквенных позициях:

- J: `Минимальная цена, RUB`
- K: `Участие товара в акции`
- L: `Итоговая цена по акции, RUB`
- O: `Цена для минимального акционного бустинга, RUB`
- P: `Цена для максимального акционного бустинга, RUB`
- R: `Остаток на складе Ozon, шт`

## 5.4. Правило приоритета буквенной позиции

Для Ozon обязательные колонки определяются строго по фактическим буквенным позициям листа.

Заголовки в строках 1–3:
- используются как дополнительная проверка корректности шаблона;
- но не заменяют первичный критерий по буквенным позициям.

## 5.5. Следствие для реализации

Запрещено строить контракт Ozon только по текстовым названиям колонок без контроля буквенных позиций.

---

## 6. Допустимость нестандартного оформления Ozon workbook

## 6.1. Что допустимо

Допустимы:
- merged cells в строках 1–3;
- нестандартное форматирование в строках 1–3;
- формулы в строках 1–3;
- прочие особенности шаблона в верхней служебной части,

если одновременно выполняется следующее:
- лист `Товары и цены` существует;
- нужные колонки присутствуют на правильных буквенных позициях;
- книга успешно читается и сохраняется без разрушения структуры;
- итоговый файл остаётся пригодным для загрузки в Ozon.

## 6.2. Что недопустимо

Если workbook читается или сохраняется так, что:
- нарушается структура книги;
- теряется пригодность к обратной загрузке;
- ломаются обязательные области шаблона;

то файл считается невалидным для обработки.

---

## 7. Формулы и пересчёт Ozon workbook

## 7.1. Правило по формулам

Формулы в книге Ozon должны сохраняться:
- без принудительного пересчёта Excel

## 7.2. Что запрещено

Запрещено:
- выполнять принудительный Excel-compatible recalc;
- менять workbook так, чтобы формулы пересчитались неуправляемо;
- разрушать формульную структуру файла.

## 7.3. Следствие для Excel writer

Выбранный writer должен:
- сохранить книгу без разрушения структуры;
- не требовать принудительного пересчёта;
- не ломать пригодность файла для загрузки.

---

## 8. Разрешённые изменения в Ozon

## 8.1. Изменяемые колонки

Разрешено менять только:
- K
- L

## 8.2. Запрещённые изменения

Запрещено изменять:
- J
- O
- P
- R
- любые другие колонки;
- структуру шаблона;
- листы книги;
- служебные области, если это не требуется технически для безопасного сохранения.

## 8.3. Минимальный критерий приёмки

Итоговый Ozon-файл должен:
- содержать изменения только в K и L;
- сохранять пригодность для обратной загрузки;
- не требовать ручного восстановления структуры.

---

## 9. Общий pipeline Ozon

## 9.1. Check pipeline

Ozon check состоит из следующих стадий:

1. Проверка наличия ровно одного файла
2. Открытие workbook
3. Проверка листа `Товары и цены`
4. Проверка обязательных колонок по буквам
5. Проверка пригодности workbook к безопасной обработке
6. Чтение строк с 4-й строки
7. Нормализация J, O, P, R
8. Определение решения по каждой строке
9. Формирование summary audit
10. Формирование detail audit
11. Формирование short_result_text
12. Завершение run без output file

## 9.2. Process pipeline

Ozon process состоит из следующих стадий:

1. Auto-validation
2. Открытие workbook
3. Проверка листа `Товары и цены`
4. Проверка обязательных колонок по буквам
5. Проверка возможности безопасного сохранения output workbook
6. Чтение строк с 4-й строки
7. Нормализация J, O, P, R
8. Определение решения по каждой строке
9. Запись только K и L
10. Сохранение output workbook
11. Формирование summary audit
12. Формирование detail audit
13. Формирование short_result_text
14. Завершение successful/failed process

---

## 10. Граница check для Ozon

## 10.1. Что check обязан делать

На этапе check для Ozon разрешено и требуется:
- проверять структуру книги;
- проверять обязательный лист и обязательные колонки;
- нормализовать значения;
- по каждой строке вычислять, попала бы она под правило:
  - missing_min_price
  - no_stock
  - no_boost_prices
  - use_max_boost_price
  - use_min_price
  - below_min_price_threshold
  - insufficient_ozon_input_data
- формировать краткий результат, сводный аудит и детальный аудит так, как если бы обработка была выполнена.

## 10.2. Что check запрещено делать

Check не должен:
- создавать output file;
- записывать значения в K и L;
- менять исходный workbook.

---

## 11. Граница process для Ozon

## 11.1. Что process обязан делать

Process обязан:
- выполнить обязательную auto-validation;
- выполнить полную построчную обработку;
- изменить только K и L;
- сохранить output workbook;
- сформировать summary audit;
- сформировать detail audit;
- сформировать short_result_text.

## 11.2. Что process запрещено делать

Process не должен:
- менять любые колонки, кроме K и L;
- разрушать workbook structure;
- изменять листы книги;
- принудительно пересчитывать формулы;
- подменять правила решения по строкам альтернативной логикой.

---

## 12. Нормализация Ozon значений

## 12.1. Какие колонки нормализуются

Нормализация применяется к значениям:
- J
- O
- P
- R

То есть к:
- Минимальная цена
- Цена для минимального бустинга
- Цена для максимального бустинга
- Остаток

## 12.2. Правила нормализации

Для J, O, P, R:
- общая нормализация чисел;
- `*` считается отсутствующим значением;
- пустая строка считается отсутствующим значением;
- `None` считается отсутствующим значением.

## 12.3. Общая числовая нормализация

Числовая нормализация должна быть согласована с общим правилом системы:
- удалить пробелы и неразрывные пробелы;
- заменить запятую на точку;
- значение, не распознанное как число, считать отсутствующим.

## 12.4. Использование нормализованных значений

После нормализации логика решения по строке должна работать только с нормализованными значениями.

---

## 13. Каноническая логика решения по строке Ozon

Ниже фиксируется обязательный и закрытый порядок правил.

## 13.1. Правило 1 — missing_min_price

Если J отсутствует:
- K = пусто
- L = пусто
- reason = `missing_min_price`

## 13.2. Правило 2 — no_stock

Если R отсутствует или <= 0:
- K = пусто
- L = пусто
- reason = `no_stock`

## 13.3. Правило 3 — no_boost_prices

Если O и P одновременно отсутствуют:
- K = пусто
- L = пусто
- reason = `no_boost_prices`

## 13.4. Правило 4 — use_max_boost_price

Если P присутствует и P >= J:
- K = `Да`
- L = P
- reason = `use_max_boost_price`

## 13.5. Правило 5 — use_min_price

Если:
- P присутствует
- O присутствует
- P < J
- O >= J

то:
- K = `Да`
- L = J
- reason = `use_min_price`

## 13.6. Правило 6 — below_min_price_threshold

Если O присутствует и O < J:
- K = пусто
- L = пусто
- reason = `below_min_price_threshold`

## 13.7. Правило 7 — insufficient_ozon_input_data

Иначе:
- K = пусто
- L = пусто
- reason = `insufficient_ozon_input_data`

---

## 14. Порядок применения правил Ozon

## 14.1. Обязательность порядка

Порядок правил из раздела 13 является обязательным.
Нельзя:
- менять условия местами;
- сначала проверять бусты, если отсутствует минимальная цена;
- сначала назначать участие, если отсутствует остаток;
- вводить дополнительные ветки между установленными правилами.

## 14.2. Почему порядок важен

Порядок определяет итоговую трактовку строки.
Изменение порядка может поменять:
- reason;
- значения K и L;
- summary counters;
- business meaning результата.

---

## 15. Правила записи значений в K и L

## 15.1. Колонка K

Если товар участвует:
- в K записывается строго строка `Да`

Если товар не участвует:
- в K записывается пустая ячейка

## 15.2. Колонка L

Если есть итоговая цена акции:
- в L записывается числовое значение

Если участия нет:
- в L записывается пустая ячейка

## 15.3. Формат числа

Числовые значения цены записываются как числа.

Целые числа:
- могут записываться без дробной части

## 15.4. Что запрещено

Запрещено:
- писать в K что-либо кроме `Да` или пустого значения;
- писать текстовые суррогаты чисел в L;
- записывать “0” вместо пустого значения там, где по ТЗ должна быть пустая ячейка.

---

## 16. Критические ошибки Ozon

## 16.1. Закрытый перечень обязательных критических ошибок

Ozon run должен завершаться failed/check_failed/validation_failed/failed при сценариях:

- файл не загружен;
- загружено более одного файла;
- книга повреждена;
- нет листа `Товары и цены`;
- отсутствуют обязательные колонки по буквам;
- невозможность сохранить выходной файл.

## 16.2. Для check

На check эти сценарии должны приводить к:
- `check_failed`

## 16.3. Для process

На process эти сценарии должны приводить к:
- `validation_failed`, если проблема выявлена на стадии validating
или
- `failed`, если ошибка произошла уже на стадии processing/save

Архитектурно предпочтительно:
- ловить максимум таких ошибок на validating.

---

## 17. Предупреждения Ozon

## 17.1. Обязательные warning scenarios

В Ozon-модуле должны быть предупреждения по следующим сценариям:
- недостаточно данных;
- отсутствует минимальная цена;
- отсутствует остаток;
- отсутствуют бусты;
- значения не распознаны как числа;
- исключение из акции по пороговой логике.

## 17.2. Связь warnings с reason

Предупреждения должны быть согласованы с reason-кодами:
- missing_min_price
- no_stock
- no_boost_prices
- below_min_price_threshold
- insufficient_ozon_input_data

## 17.3. Влияние warnings на run outcome

Наличие warning rows:
- не обязательно делает весь run failed;
- но может привести к:
  - check_passed_with_warnings
  - completed_with_warnings

---

## 18. Summary audit Ozon

## 18.1. Обязательные показатели

Ozon summary audit обязан включать:
- всего строк данных;
- успешно обработано;
- предупреждения;
- участвуют;
- не участвуют;
- нет минимальной цены;
- нет остатка;
- нет бустов;
- use_max_boost_price;
- use_min_price;
- below_min_price_threshold;
- изменено строк.

## 18.2. Для check и process

Этот набор должен быть доступен и для:
- check
- process

Потому что check моделирует итог так, как если бы обработка была выполнена.

## 18.3. Роль summary

Summary должен позволять быстро понять:
- сколько строк участвует;
- сколько не участвует;
- по каким основным причинам строки исключаются;
- сколько строк реально изменится в process.

---

## 19. Short result Ozon

## 19.1. Обязательный состав

Краткий результат Ozon должен включать:
- изменено строк;
- участвуют;
- не участвуют;
- предупреждения;
- при ошибке — короткую причину.

## 19.2. Назначение short result

Short result используется для:
- history list
- dashboard
- run header
- polling result after completion

Он не заменяет summary audit.

---

## 20. Detail audit Ozon

## 20.1. Обязательные колонки UI

Для Ozon detail audit обязательны:
- row_number
- OzonID/SKU/Артикул
- min_price
- min_boost
- max_boost
- stock
- final_participation
- final_promo_price
- severity
- decision_reason
- message

## 20.2. Источник данных detail audit

Эти поля должны формироваться из:
- общих detail audit полей
- Ozon audit payload

UI не должен перечитывать workbook для их вывода.

## 20.3. Правило entity_key_1

В `entity_key_1` должен использоваться:
- OzonID/SKU/Артикул  
в приоритетном порядке доступности.

## 20.4. Severity semantics по строкам

### `info`
Нормальная строка с однозначно определённым решением.

### `warning`
Строка обработана, но имеются отклонения:
- отсутствуют данные;
- бусты отсутствуют;
- остаток отсутствует;
- строка исключена по логике threshold/min-price;
- значения не распознаны как числа.

### `error`
Строка содержит ошибку уровня строки, если это требуется прикладной моделью для отражения некорректности, не делающей run целиком failed.

Важно:
- итог run определяется не одной detail row, а всей совокупностью обработки и критическими ошибками запуска.

---

## 21. Рекомендуемый набор Ozon decision_reason

ТЗ задаёт ключевые reason-коды прямо в бизнес-логике.
Для архитектурной стабильности Ozon-модуль должен использовать следующий стандартизированный набор:

- missing_min_price
- no_stock
- no_boost_prices
- use_max_boost_price
- use_min_price
- below_min_price_threshold
- insufficient_ozon_input_data
- invalid_numeric_value

Это не расширяет ТЗ, а стандартизирует detail audit explainability.

---

## 22. Check outcome mapping для Ozon

## 22.1. `check_passed`

Используется, если:
- критических ошибок запуска нет;
- warning-level отклонения отсутствуют или не приводят к итоговому предупреждению запуска.

## 22.2. `check_passed_with_warnings`

Используется, если:
- check выполнен;
- есть warning-level строки;
- обработка в целом остаётся валидной как проверка.

## 22.3. `check_failed`

Используется, если:
- возникла критическая ошибка запуска;
- нельзя корректно выполнить Ozon check.

---

## 23. Process outcome mapping для Ozon

## 23.1. `validation_failed`

Используется, если на стадии auto-validation выявлено:
- файл не загружен;
- загружено более одного файла;
- книга повреждена;
- нет листа `Товары и цены`;
- отсутствуют обязательные колонки по буквам;
- workbook не может быть безопасно сохранён как корректный output.

## 23.2. `completed`

Используется, если:
- process завершён успешно;
- output file сформирован;
- warning-level итоговых отклонений нет.

## 23.3. `completed_with_warnings`

Используется, если:
- process завершён успешно;
- output file сформирован;
- есть warning-level отклонения по строкам.

## 23.4. `failed`

Используется, если:
- processing/save сорвались уже после стадии принятия и/или validating.

---

## 24. Связь Ozon detail audit с summary counters

Summary audit Ozon должен агрегироваться из того же canonical processing result, который формирует detail audit.

Нельзя допускать расхождение, когда:
- detail rows показывают один набор reasons;
- summary считает другое распределение причин;
- short result сообщает несовместимые цифры.

Следовательно:
- summary builder и detail builder должны работать от единого доменного результата.

## 25. Каноническая последовательность обработки строки Ozon

Для каждой строки начиная с 4-й канонический порядок должен быть таким:

1. Определить row_number
2. Считать идентификатор строки для entity_key_1
3. Нормализовать J
4. Нормализовать O
5. Нормализовать P
6. Нормализовать R
7. Проверить правило missing_min_price
8. Проверить правило no_stock
9. Проверить правило no_boost_prices
10. Проверить правило use_max_boost_price
11. Проверить правило use_min_price
12. Проверить правило below_min_price_threshold
13. Иначе применить insufficient_ozon_input_data
14. Сформировать detail audit row
15. Учесть строку в summary counters
16. При process — записать K и L

Изменять этот порядок нельзя, потому что он определяет логику результата.

## 26. Workbook save contract для Ozon

## 26.1. Требование сохранности структуры

Результирующий Ozon workbook должен:
- сохранять структуру книги;
- сохранять пригодность для обратной загрузки;
- не ломать шаблон.

## 26.2. Минимальный обязательный критерий приёмки

Минимальный обязательный критерий:
- изменяются только K и L;
- структура книги не ломается;
- результат пригоден для обратной загрузки.

## 26.3. Следствие для реализации

Выбранный writer для Ozon должен быть подобран под safe-save сценарий.
Нельзя выбирать реализацию, которая:
- ломает merged/header structure;
- нарушает шаблон;
- делает файл непригодным для загрузки в Ozon.
