# 14_RISKS_LIMITS_AND_CONTROL_POINTS.md

## 1. Назначение документа

Документ фиксирует:
- критические риски проекта;
- жёсткие ограничения MVP;
- точки архитектуры и реализации, где особенно важно не нарушить ТЗ;
- обязательные контрольные запреты;
- проектные уязвимости, требующие отдельного внимания;
- контрольные вопросы для оркестратора, разработчика, аудитора и тестировщика.

Документ не вводит новые требования сверх ТЗ.
Его задача — зафиксировать, где именно проект чаще всего может быть реализован неправильно.

---

## 2. Краткий итог

У проекта есть несколько зон максимального риска:

1. нарушение границ MVP  
2. смешение check и process  
3. смешение WB и Ozon логики  
4. неверная модель доступа  
5. синхронная долгая обработка вместо run + polling  
6. неправильная работа с Excel и разрушение файлов  
7. неправильная файловая модель tmp/input/output  
8. отсутствие реальной блокировки store+module  
9. некорректный audit/logging contour  
10. неполная server-side модель history/logs/detail audit  

Самые опасные дефекты для приёмки:
- ломается workbook WB или Ozon;
- process запускается/работает не по ТЗ;
- roles/permissions/store access соблюдаются неполно;
- detail audit не хранится в PostgreSQL;
- появляются лишние сущности и сценарии вне MVP;
- допускаются два активных run по одной паре store+module;
- используется float в финальном расчёте WB.

## 3. Базовые ограничения, которые нельзя нарушать

## 3.1. Ограничения scope MVP

Строго запрещено добавлять в MVP:
- API маркетплейсов;
- API-ключи;
- периодические фоновые запуски как пользовательскую функцию;
- уведомления;
- telegram-боты;
- внешние интеграции;
- аналитику;
- массовые очереди для разных подсистем;
- recovery/reset password by email;
- поддержку форматов кроме `.xlsx`;
- экспорт detail audit отдельным файлом.

## 3.2. Ограничения модели данных

Запрещено:
- расширять список обязательных бизнес-сущностей без реальной технической необходимости;
- заменять run/audit/log/file модель упрощённым вариантом;
- убирать одну из обязательных таблиц;
- выносить detail audit только в файлы.

## 3.3. Ограничения выполнения

Запрещено:
- делать долгую синхронную обработку через HTTP;
- заменять async execution на “подождать результат в запросе”;
- убирать polling;
- разрешать два активных run по одной паре store+module.

## 3.4. Ограничения бизнес-логики

Запрещено:
- менять формулу WB;
- менять порядок условий WB fallback logic;
- менять порядок правил Ozon;
- изменять закрытые наборы lifecycle_status и business_result;
- вводить альтернативные трактовки check/process.

## 4. Риск №1. Нарушение границ MVP

## 4.1. Суть риска

При проектировании и реализации есть соблазн:
- заложить внешние интеграции “на будущее”;
- добавить не требуемые страницы;
- расширить админку;
- ввести лишние пользовательские сценарии.

Это создаёт scope creep и размывает приёмку.

## 4.2. Признаки риска

- появляются сущности API keys;
- появляются разделы интеграций;
- добавляются фоновые расписания как пользовательская функция;
- появляется экспорт detail audit в отдельный файл;
- добавляется analytics dashboard beyond MVP;
- добавляются уведомления или email flows.

## 4.3. Контрольная точка

На любом review нужно проверять:
- реализует ли change что-то, чего нет в ТЗ;
- не появился ли скрытый нефункциональный scope как новая функция.

## 4.4. Что считается нарушением

Нарушением считается даже “подготовка” пользовательского интерфейса под API keys/integrations,
если это уже проявляется как сущность, страница, настройка или новый workflow в MVP-сборке.

---

## 5. Риск №2. Смешение check и process

## 5.1. Суть риска

Очень частая ошибка — реализовать process как:
- “тот же check, только потом записать файл”
или
- общий universal handler c dry_run flag.

Это нарушает ТЗ, потому что:
- check и process — самостоятельные операции;
- у них разные lifecycle paths;
- у process есть обязательная validating phase;
- check не создаёт output file.

## 5.2. Признаки риска

- один handler выполняет и check, и process с флагом;
- validating phase для process отсутствует;
- check и process делят одну history semantics;
- process не имеет отдельного business_result path.

## 5.3. Контрольная точка

Нужно отдельно проверять:
- есть ли отдельные use cases;
- есть ли отдельные lifecycle transitions;
- есть ли отдельный execution path для process;
- отсутствует ли output file у check.

## 5.4. Критический запрет

Нельзя допускать архитектуру, где check и process различаются только наличием/отсутствием сохранения workbook в самом конце.

---

## 6. Риск №3. Смешение WB и Ozon логики

## 6.1. Суть риска

Опасно строить “общий движок маркетплейса” и пытаться через конфигурацию подмешивать WB и Ozon правила.

Это приводит к:
- расползанию условий;
- неявным зависимостям;
- риску испортить один модуль при доработке другого.

## 6.2. Признаки риска

- общий marketplace engine;
- общий workbook processor;
- общие rule chains с динамическими ветками;
- WB/Ozon detail audit строятся одним и тем же неразделённым кодом.

## 6.3. Контрольная точка

Нужно отдельно проверять:
- разделены ли wb и ozon модули;
- разделены ли workbook readers/writers;
- разделены ли summary/detail audit builders;
- нет ли hidden cross-dependency.

## 6.4. Критический запрет

Любая реализация, где Ozon и WB живут в одном условном “универсальном Excel processor” без жёсткой модульной изоляции, считается архитектурно опасной.

---

## 7. Риск №4. Нарушение модели доступа

## 7.1. Суть риска

Система использует трёхкомпонентную модель доступа:
- role
- permissions
- user_store_access

Частая ошибка — реализовать только часть из них.

## 7.2. Признаки риска

- доступ определяется только ролью;
- permissions не проверяются в application layer;
- store access проверяется только в UI;
- admin override не реализован отдельно;
- history/logs/run pages не ограничены по store scope.

## 7.3. Контрольная точка

Нужно отдельно проверять:
- page visibility;
- action authorization;
- read-side scoping;
- file download authorization;
- no-store scenario;
- archived store restrictions.

## 7.4. Критические последствия

Ошибки в access model ведут к:
- утечке чужих запусков;
- скачиванию чужих файлов;
- видимости чужой истории;
- нарушению приёмки по security/access.

---

## 8. Риск №5. Синхронная обработка вместо run + polling

## 8.1. Суть риска

Из-за простоты реализации может возникнуть соблазн:
- выполнять check/process прямо в HTTP;
- держать запрос до результата;
- не делать реальный background execution.

Это прямое нарушение ТЗ.

## 8.2. Признаки риска

- POST /check или /process ждёт полный результат;
- polling endpoint формальный и не нужен реально;
- lifecycle_status не успевает пройти промежуточные состояния;
- run создаётся только “для истории”, а не как реальный execution object.

## 8.3. Контрольная точка

Нужно проверять:
- acceptance response <= целевого времени;
- lifecycle transitions наблюдаемы;
- polling реально нужен;
- execution происходит вне HTTP.

## 8.4. Критический запрет

Нельзя маскировать синхронную обработку под pseudo-async, где run создаётся, но выполнение всё равно происходит в том же запросе.

---

## 9. Риск №6. Отсутствие реальной блокировки store+module

## 9.1. Суть риска

Если не реализовать настоящий lock/conflict control, система может допустить:
- два process по одному store;
- check и process одновременно;
- две competing записи результата.

## 9.2. Признаки риска

- active run conflict проверяется только на UI;
- нет повторной проверки на execution layer;
- два параллельных запроса могут пройти race condition;
- superseded logic используется как замена lock, а не как аварийный fallback.

## 9.3. Контрольная точка

Нужно отдельно проверять:
- pre-run conflict detection;
- lock acquisition before execution;
- lock release after success/failure/timeout;
- repeated conflict checks in race scenarios.

## 9.4. Критический запрет

Нельзя считать проблему решённой только потому, что UI “не даёт нажать кнопку дважды”.

---

## 10. Риск №7. Неправильная модель временных файлов

## 10.1. Суть риска

Легко неверно реализовать active temp set:
- допустить несколько активных наборов;
- использовать старые файлы без явного active set;
- не считать новую загрузку новым временным объектом;
- не очищать temp files по TTL.

## 10.2. Признаки риска

- у user+store+module одновременно несколько независимых активных наборов;
- tmp files напрямую считаются history;
- re-upload same file не инвалидирует старый check;
- TTL purge отсутствует.

## 10.3. Контрольная точка

Нужно проверять:
- один активный набор на user+store+module;
- delete/replace semantics;
- invalidation previous check by any composition change;
- purge after 24h;
- empty set after purge.

## 10.4. Критический запрет

Нельзя использовать temporary storage как единственный источник исторических run input files.

---

## 11. Риск №8. Неверная файловая модель history

## 11.1. Суть риска

Если run input files не копируются физически,
история становится невоспроизводимой после очистки temp files.

## 11.2. Признаки риска

- run ссылается на tmp path;
- one physical file reused across runs;
- input copies deduplicated between runs;
- output files хранятся в том же контуре, что tmp uploads.

## 11.3. Контрольная точка

Нужно проверять:
- есть ли физические копии input files per run;
- разделены ли tmp / input / output paths;
- сохраняется ли run после purge tmp files;
- metadata и filesystem согласованы ли.

## 11.4. Критический запрет

Дедупликация input files по хэшу между разными runs в MVP запрещена и не должна появиться даже как “оптимизация”.

---

## 12. Риск №9. Разрушение Excel-файлов WB

## 12.1. Суть риска

WB-файл особенно чувствителен к неправильной записи:
- можно случайно испортить структуру книги;
- потерять стили/формулы;
- изменить лишние колонки;
- сделать workbook непригодным.

## 12.2. Признаки риска

- меняются колонки кроме `Новая скидка`;
- перестраивается workbook целиком;
- пропадают формулы;
- ломаются скрытые элементы, фильтры, ширины;
- unsafe write не детектируется заранее.

## 12.3. Контрольная точка

Нужно проверять:
- только `Новая скидка` меняется;
- row order preserved;
- workbook remains usable;
- unsafe write correctly blocks process.

## 12.4. Особо опасное место

Колонка `Новая скидка` — отдельная зона риска.
Если она:
- защищена;
- формульная;
- нестандартно форматирована;
- unsafe to write,

система должна остановиться, а не пытаться сохранить файл любой ценой.

---

## 13. Риск №10. Разрушение Excel-файлов Ozon

## 13.1. Суть риска

Ozon шаблон чувствителен к:
- структуре листа;
- колонкам по буквам;
- шапке;
- формульным и служебным областям;
- пригодности для обратной загрузки.

## 13.2. Признаки риска

- колонки определяются не по буквам, а только по тексту;
- меняются колонки кроме K/L;
- forced recalculation применяется;
- нарушается структура шапки/служебной области;
- итоговый файл не загружается обратно в Ozon.

## 13.3. Контрольная точка

Нужно проверять:
- обязательный лист;
- J/K/L/O/P/R positions;
- start row = 4;
- only K/L changed;
- workbook remains uploadable.

## 13.4. Критический запрет

Нельзя применять библиотеку/режим сохранения, который ломает Ozon template, даже если “данные формально записались”.

---

## 14. Риск №11. Использование float в WB

## 14.1. Суть риска

Это один из самых критичных и проверяемых дефектов.
Если в финальном расчёте WB использовать float:
- возможны ошибки округления;
- изменится итоговая скидка;
- система нарушит ТЗ.

## 14.2. Признаки риска

- в calculation path используются float/double;
- формула строится на стандартных вещественных операциях;
- ceil применяется к результату float-погрешности.

## 14.3. Контрольная точка

Нужно отдельно проверять:
- типы в расчёте;
- decimal-safe division;
- decimal-safe multiplication;
- final integer conversion with ceil.

## 14.4. Критический запрет

Даже частичное использование float в промежуточной финальной ветке WB-формулы считается нарушением ТЗ.

---

## 15. Риск №12. Неверный порядок условий WB и Ozon

## 15.1. Суть риска

В обоих модулях порядок условий определяет бизнес-результат.

Если менять порядок:
- изменится reason;
- изменятся итоговые counters;
- изменится final result.

## 15.2. Для WB

Особенно опасно перепутать:
- promo_not_found
- threshold branch
- ordinary branch

## 15.3. Для Ozon

Особенно опасно перепутать:
- missing_min_price
- no_stock
- no_boost_prices
- use_max_boost_price
- use_min_price
- below_min_price_threshold
- insufficient_ozon_input_data

## 15.4. Контрольная точка

Нужно проверять:
- exact branching order;
- no hidden shortcut logic;
- no rule merging “для упрощения”.

---

## 16. Риск №13. Неполный или несогласованный audit

## 16.1. Суть риска

Можно реализовать:
- summary audit без detail;
- detail audit без full required payload;
- inconsistent short_result vs summary vs business_result;
- audit, требующий перечитывать workbook.

Это нарушит explainability и приёмку.

## 16.2. Признаки риска

- detail audit rows отсутствуют при completed run;
- summary counters не покрывают обязательные показатели;
- short_result текст не согласован с result;
- run page требует заново читать workbook для нужных колонок.

## 16.3. Контрольная точка

Нужно проверять:
- для любого завершённого run есть summary + detail;
- required module-specific columns доступны из audit;
- short_result, summary и business_result согласованы.

## 16.4. Критический запрет

Нельзя хранить detail audit только во внешнем JSON/Excel и считать это “достаточным”.

---

## 17. Риск №14. Смешение system logs и run audit

## 17.1. Суть риска

Разработчики часто пытаются заменить:
- logs -> audit
или
- audit -> logs

Это ломает и observability, и UI.

## 17.2. Признаки риска

- history строится из logs;
- run page строится из logs вместо audit;
- detail audit складывается в system_logs;
- logs page показывает detail rows как system events.

## 17.3. Контрольная точка

Нужно отдельно проверять:
- run page source
- logs page source
- history source
- detail audit storage source

## 17.4. Критический запрет

System logs и run audit — разные контуры, их нельзя сводить к одной сущности “для простоты”.

---

## 18. Риск №15. Client-side heavy loading вместо server-side read model

## 18.1. Суть риска

Из-за удобства реализации можно попытаться:
- вытягивать весь history;
- вытягивать весь logs table;
- вытягивать весь detail audit и фильтровать в UI.

Это нарушает ТЗ и делает систему неустойчивой.

## 18.2. Признаки риска

- UI получает полную выборку и фильтрует сам;
- pagination client-side;
- search выполняется в браузере;
- detail audit 200k rows загружается целиком.

## 18.3. Контрольная точка

Нужно проверять:
- search/filter/sort/page реально исполняются на сервере;
- UI не получает полный массив;
- large datasets не загружаются целиком.

## 18.4. Критический запрет

Любая client-side pagination/filtering для history/logs/detail audit как основной способ работы — нарушение ТЗ.

---

## 19. Риск №16. Неправильная работа с archived stores

## 19.1. Суть риска

Archived store имеет двойственную природу:
- сохраняется в истории;
- запрещён для новых запусков;
- не редактируется;
- может быть восстановлен только admin.

Частая ошибка — либо полностью скрыть архивный store, либо продолжить разрешать операции.

## 19.2. Контрольная точка

Нужно проверять:
- archived store видим в history context;
- не попадает в selector новых запусков;
- не редактируется;
- restore only by admin.

---

## 20. Риск №17. Неполная реализация no-store user scenario

## 20.1. Суть риска

ТЗ явно определяет поведение пользователя без магазинов.
Если пропустить этот сценарий, интерфейс и доступы будут реализованы неверно.

## 20.2. Контрольная точка

Нужно проверять:
- пользователь входит;
- видит dashboard;
- видит “Нет доступных магазинов”;
- не видит history;
- не запускает processing;
- manager_lead с create_store видит возможность создать магазин.

---

## 21. Риск №18. Неполная реализация retention и superseded

## 21.1. Суть риска

Можно реализовать happy-path обработки, но не довести:
- temp purge;
- run file expiration;
- superseded result;
- timeout cleanup;
- anomaly logging.

Такой проект формально работает, но не проходит по эксплуатационной устойчивости.

## 21.2. Контрольная точка

Нужно отдельно проверять:
- temp purge after 24h;
- file availability markers;
- superseded old outputs;
- timeout finalization;
- anomaly logs.

---

## 22. Особо критичные инварианты проекта

Ниже перечислены инварианты, нарушение которых почти гарантированно означает отклонение от ТЗ.

1. Все обязательные сущности реализованы.
2. Roles + permissions + user_store_access работают вместе.
3. Check и process — отдельные операции.
4. Process всегда имеет validating phase.
5. Нет двух активных runs по одной паре store+module.
6. Polling обязателен.
7. Detail audit полностью хранится в PostgreSQL.
8. WB использует decimal, а не float.
9. WB меняет только `Новая скидка`.
10. Ozon меняет только K и L.
11. Ozon columns определяются по буквам, а не только по header text.
12. Temporary files и run files разделены.
13. Run input files сохраняются как отдельные физические копии.
14. Недоступные файлы отражаются в UI корректно.
15. Logs и audit не смешиваются.
16. History/logs/detail audit используют server-side controls.
17. Archived store не участвует в новых запусках.

## 27. Что нельзя упрощать “ради скорости”

Нельзя упрощать следующие вещи даже временно:

- async execution model
- validating phase process
- store+module locking
- temp vs run files separation
- secure download flow
- detail audit in PostgreSQL
- access checks in application layer
- WB exact arithmetic
- Ozon column-by-letter validation
- server-side history/logs/detail audit
- superseded/retention handling
- no-store and archived-store scenarios

Любое такое “временное упрощение” почти наверняка превратится в дефект архитектуры.
