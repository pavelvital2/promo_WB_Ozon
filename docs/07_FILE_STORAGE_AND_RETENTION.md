# 07_FILE_STORAGE_AND_RETENTION.md

## 1. Назначение документа

Документ фиксирует:
- файловую архитектуру системы;
- разделение временных файлов и run-файлов;
- структуру хранения на файловой системе VPS;
- правила формирования активного временного набора;
- правила сохранения input files per run;
- правила сохранения output files per run;
- retention rules;
- причины недоступности файлов;
- superseded handling;
- правила UI-поведения при недоступных файлах;
- обязательные системные события, связанные с файлами.

Документ не описывает:
- жизненный цикл run как таковой;
- UI-маршруты;
- логику расчёта WB/Ozon;
- SQL-структуру таблиц beyond already fixed schema.

---

## 2. Краткий итог

Файловое хранение системы разделяется на три независимых контура:

1. temporary uploads  
2. run input files  
3. run output files  

Ключевые правила:
- временные файлы живут отдельно от истории запусков;
- для каждого run входные файлы сохраняются как отдельные физические копии;
- дедупликация файлов между разными runs запрещена;
- successful process создаёт отдельный output file;
- check output file не создаёт;
- файлы не доступны по прямому URL;
- недоступность файла должна быть отражена и в БД, и в UI;
- старый успешный результат при появлении нового успешного результата помечается как недоступный, но historical run сохраняется;
- временные файлы автоматически очищаются через 24 часа, если не были использованы как часть run.

---

## 3. Общая файловая модель

## 3.1. Файловые контуры

В системе существуют три типа файлового контекста:

### Temporary uploads
Файлы, загруженные пользователем до запуска check/process.

### Run input files
Исторические входные файлы, зафиксированные как часть конкретного run.

### Run output files
Исторические выходные файлы конкретного successful process-run.

Эти три контура не должны смешиваться ни физически, ни логически.

## 3.2. Почему разделение обязательно

Разделение нужно для соблюдения ТЗ:
- активный временный набор может изменяться;
- исторические input files должны быть воспроизводимыми;
- output files должны быть отдельными артефактами результата;
- check/process работают по текущему активному набору, но история должна хранить точную копию фактически использованных файлов;
- retention и недоступность применяются по разным правилам к разным контурам.

---

## 4. Физическое место хранения

## 4.1. Общее требование

Все файлы хранятся в файловой системе VPS в выделенном каталоге приложения.

Прямой публичный доступ по открытому URL запрещён.

## 4.2. Обязательная логическая структура

Минимально обязательная логическая структура:

- `/storage/uploads/tmp/`
- `/storage/runs/{module_code}/{store_id}/{public_run_number}/input/`
- `/storage/runs/{module_code}/{store_id}/{public_run_number}/output/`

## 4.3. Допустимые уточнения физической структуры

Допускается техническое уточнение вложенных каталогов, если не меняется базовая логика ТЗ.

Допустимые технические уточнения:
- вложение по дате;
- дополнительный уровень namespace;
- хэш-папки для распределения файлов.

Недопустимо:
- смешивать tmp и runs;
- смешивать input и output;
- убирать из run path связку module_code/store_id/public_run_number;
- делать физическую структуру непрозрачной для воспроизводимости run.

---

## 5. Temporary uploads

## 5.1. Назначение временных файлов

Temporary uploads нужны только для пользовательской подготовки текущего активного набора файлов перед запуском check/process.

Они используются для:
- показа загруженных файлов на странице обработки;
- удаления файла из набора;
- замены файла;
- запуска check/process по текущему набору.

## 5.2. Контур временного набора

Временный набор существует только в контуре:
- user
- store
- module

У одного пользователя для одной пары store + module:
- существует только один текущий активный временный набор.

Поддержка нескольких параллельных активных наборов в рамках одной пары:
- запрещена.

## 5.3. Состав активного набора

Активный набор — это все записи temporary_uploaded_files, которые одновременно принадлежат:
- одному `uploaded_by_user_id`
- одному `store_id`
- одному `module_code`
- и имеют `is_active_in_current_set = true`

## 5.4. Что пользователь должен видеть

На страницах обработки пользователь должен видеть текущий активный временный набор с возможностью:
- увидеть имя файла;
- увидеть размер файла;
- увидеть время загрузки;
- удалить файл из набора;
- заменить файл.

## 5.5. Изменение состава файлов

Любое из перечисленного меняет состав набора:
- удаление одного файла;
- добавление одного файла;
- замена файла;
- повторная загрузка файла с тем же именем, но иным содержимым;
- повторная загрузка файла с тем же содержимым, но иным именем;
- повторная загрузка файла с тем же именем и тем же содержимым, если создан новый временный объект.

Следствие:
- предыдущий check по старому набору становится неактуальным.

---

## 6. Идентичность файла и загрузка нового временного объекта

## 6.1. Признак идентичности содержимого

Файл считается “тем же самым” по содержимому только по комбинации:
- SHA-256
- размер в байтах

Оригинальное имя файла не является достаточным признаком идентичности.

## 6.2. Важное ограничение

Даже если:
- имя совпадает;
- SHA-256 совпадает;
- размер совпадает;

но пользователь загрузил файл заново и был создан новый temporary_uploaded_file object,
то это считается новым элементом временного контура.

Следствие:
- старый check всё равно становится недействительным.

---

## 7. Временные файлы: сроки хранения и очистка

## 7.1. Срок хранения

Срок хранения временного файла:
- 24 часа с момента загрузки

## 7.2. Что подлежит автоматической очистке

Автоматической очистке подлежат:
- временные файлы, не использованные для запуска;
- временные файлы, у которых истёк TTL;
- записи о них в temporary_uploaded_files, если принята стратегия физического удаления и удаления metadata временного контура.

Это допустимо, потому что ТЗ прямо разрешает физическое удаление временных технических файлов и записей о них.

## 7.3. Что происходит после очистки

Если пользователь открывает экран после очистки:
- набор считается пустым

Отдельная UI-индикация факта автоматической очистки:
- не требуется

## 7.4. Логирование очистки

Автоматическая очистка временных файлов должна журналироваться как системное событие.

Минимальный event_type:
- `temporary_files_auto_purged`

---

## 8. Run input files

## 8.1. Назначение

Run input files — это исторические копии фактически использованных входных файлов конкретного run.

## 8.2. Когда создаются

Run input files создаются при фактическом запуске check/process:
- после принятия run;
- в рамках execution path;
- до прикладной обработки соответствующего модуля;
- из текущего активного временного набора на момент запуска.

## 8.3. Обязательное правило физических копий

Для каждого run входные файлы должны сохраняться как отдельные физические копии.

Дедупликация файлов по хэшу между разными runs:
- запрещена.

Следствие:
- даже если пользователь дважды запускает process по одному и тому же набору,
каждый run должен иметь свои собственные физические input copies.

## 8.4. Каталог хранения

Run input files должны храниться в каталоге:
- `/storage/runs/{module_code}/{store_id}/{public_run_number}/input/`

## 8.5. Что должно храниться по каждому input file

Обязательно должны храниться:
- original_filename
- stored_filename
- storage_relative_path
- mime_type
- file_size_bytes
- file_sha256
- uploaded_at_utc
- expires_at_utc
- доступность / недоступность файла

## 8.6. Почему input files не должны ссылаться на tmp-файлы напрямую

Нельзя считать temporary file историческим input file только по ссылке на tmp path, потому что:
- tmp-файлы очищаются по TTL;
- active set изменяемый;
- historical run должен быть воспроизводим;
- tmp contour не является history contour.

Следовательно:
- при запуске run must create dedicated copies.

---

## 9. Run output files

## 9.1. Назначение

Run output file — это исторический результат successful process-run.

## 9.2. Когда output file должен существовать

Output file создаётся только для successful process-run:
- `business_result = completed`
- `business_result = completed_with_warnings`

## 9.3. Когда output file не должен существовать

Output file не должен создаваться:
- для check;
- для process с `validation_failed`;
- для process с `failed`.

## 9.4. Каталог хранения

Run output files должны храниться в:
- `/storage/runs/{module_code}/{store_id}/{public_run_number}/output/`

## 9.5. Один результат на один successful process

Для одного successful process-run допустим не более одного логического output file результата.

Это означает:
- один `result_file_id` в runs
- одна основная выходная книга результата

---

## 10. Stored filename и path strategy

## 10.1. Stored filename

`stored_filename` должен быть:
- уникальным глобально;
- независимым от пользовательского оригинального имени;
- пригодным для безопасного хранения на filesystem.

Рекомендуемый подход:
- deterministic-safe or generated unique filename with extension preservation

Обязательное требование:
- уникальность обеспечивается системой, а не пользователем.

## 10.2. Original filename

`original_filename` хранится для:
- отображения пользователю;
- history;
- audit/context;
- удобного скачивания и трассируемости.

## 10.3. Storage relative path

`storage_relative_path` хранит путь относительно корня storage.

Он нужен для:
- безопасного поиска файла внутри application;
- отсутствия зависимости от абсолютного пути сервера;
- portability внутри deployment environment.

---

## 11. MIME, size, hash

## 11.1. MIME type

Для каждого файла хранится mime_type.
Он нужен для:
- диагностики;
- secure download response;
- валидации ожидаемого типа файла.

## 11.2. File size bytes

Хранится фактический размер файла в байтах.
Используется для:
- limit checks;
- identity rules;
- diagnostics;
- retention/reporting.

## 11.3. File SHA-256

Хранится для:
- identity verification;
- reproducibility;
- diagnostics;
- traceability.

Для tmp и run files hash должен считаться независимо для фактически сохранённого физического файла.

## 12. Файловые лимиты

## 12.1. Максимальный размер одного файла

Максимальный размер одного .xlsx файла:
- 25 МБ

Это правило применяется при загрузке temporary files.

## 12.2. Максимальный суммарный объём одного запуска WB

Для одного запуска WB:
- максимум 100 МБ суммарно по всем файлам набора

Проверка должна применяться до запуска run.

## 12.3. Максимальное количество promo files для WB

Для WB:
- от 1 до 20 файлов акций

## 12.4. Ozon

Для Ozon:
- ровно 1 файл .xlsx

## 12.5. Реакция на нарушение лимитов

При нарушении лимитов должен возвращаться структурированный отказ:
- `error_code = file_limit_exceeded`

## 13.2. Допустимые роли для Ozon

- ozon_input

Для WB run:

## 13.4. Инварианты состава файлов Ozon

- для check: ozon_result_output отсутствует

## 14. Правила доступности run files

## 14.1. Общий принцип

Run files могут быть:
- доступны
- недоступны

Доступность определяется через:
- `is_available`
- `unavailable_reason`

## 14.2. Когда файл считается доступным

Файл считается доступным, если:
- metadata запись существует;
- `is_available = true`
- физический файл существует;
- у пользователя есть право доступа к соответствующему store/run.

## 14.3. Когда файл считается недоступным

Файл считается недоступным, если:
- срок хранения истёк;
- файл удалён по retention rule;
- файл был superseded;
- запись специально переведена в unavailable по согласованному правилу хранения.

## 14.4. Причины недоступности

Закрытый набор:
- expired
- superseded
- deleted_by_retention_rule

---

## 15. Superseded results

## 15.1. Когда применяется superseded

Если по одной и той же паре:
- store_id
- module_code

появляется новый успешный process result, то предыдущий успешный result должен стать недоступным.

## 15.2. Что именно должно произойти

Система обязана:
1. сохранить новый successful run;
2. определить предыдущий успешный result по той же паре;
3. пометить старый output file как:
   - `is_available = false`
   - `unavailable_reason = superseded`
4. сохранить historical run;
5. записать системный лог.

## 15.3. Что не должно происходить

Не должно происходить:
- удаление historical run;
- удаление run_summary_audit;
- удаление run_detail_audit;
- потеря истории предыдущего запуска.

## 15.4. Гонка по текущему успешному результату

Если из-за технической ошибки всё же возникли два competing successful process-run:
- текущим успешным результатом считается тот, у которого later finished_at_utc
- другой result output file становится superseded
- в system_logs пишется ошибка согласованности

---

## 16. Expiration и retention для run files

## 16.1. Общее правило

ТЗ требует:
- хранение исходных файлов и файлов результатов по правилам;
- отражение недоступности файлов в UI;
- сохранение истории запусков;
- сохранение metadata файлов даже при недоступности.

Следовательно:
- историческая запись файла остаётся;
- физический файл может стать недоступным;
- UI обязан корректно это отразить.

## 16.2. Источники недоступности

Недоступность run file может наступить по одному из сценариев:
1. superseded
2. expired
3. deleted_by_retention_rule

## 16.3. Expiration metadata

Поле `expires_at_utc` используется как граница срока доступности файла.

Если срок хранения истёк:
- файл должен считаться недоступным;
- скачивание запрещается;
- UI показывает “Файл недоступен”.

## 16.4. Retention execution

Retention execution должен:
- находить файлы, срок хранения которых истёк;
- удалять физические файлы по принятому правилу;
- обновлять metadata:
  - `is_available = false`
  - `unavailable_reason = expired` или `deleted_by_retention_rule` в зависимости от принятой трактовки события
- писать system log.

## 16.5. Рекомендуемая строгая трактовка причин

Чтобы избежать смешения двух смыслов, рекомендуется следующее проектное правило:

- `expired`  
  использовать, когда файл уже считается просроченным для пользователя по сроку хранения

- `deleted_by_retention_rule`  
  использовать, когда retention job уже физически удалил файл

Это не меняет ТЗ, а только делает поведение прозрачным.

---

## 17. UI-поведение при недоступных файлах

## 17.1. Обязательное правило

Если срок хранения файла истёк или файл удалён по правилам хранения:
- в карточке запуска и history должна отображаться надпись:
  - “Файл недоступен”
- кнопка скачивания должна быть disabled
- у файла должен отображаться отдельный признак недоступности

## 17.2. Где это должно отражаться

Должно отражаться:
- на странице run/detail
- в history, если там есть файловые действия
- в любом месте UI, где доступен download action

## 17.3. Что не требуется

Не требуется:
- отдельная специальная страница причин недоступности;
- отдельные всплывающие уведомления о retention cleanup;
- отдельная UI-индикация факта auto-purge временных файлов.

---

## 18. Secure download model

## 18.1. Общий принцип

Ни один файл не должен скачиваться по прямому файловому URL.

Каждое скачивание должно идти через backend use case, который проверяет:
1. авторизацию;
2. user blocked state;
3. access к store/run;
4. доступность metadata;
5. `is_available = true`;
6. существование физического файла.

## 18.2. Что происходит при попытке скачать недоступный файл

Если файл:
- недоступен по metadata
или
- отсутствует физически при `is_available=true` вследствие несогласованности,

скачивание не должно происходить.
Система должна вернуть контролируемую ошибку и, при необходимости, залогировать системную несогласованность.

---

## 19. Взаимосвязь temp files и run files

## 19.1. До запуска

Работа идёт только с temporary uploads.

## 19.2. Во время запуска

Execution layer должен:
- прочитать активный набор;
- сделать физические копии в run/input;
- только после этого считать run воспроизводимым по входным данным.

## 19.3. После запуска

Temporary files:
- могут продолжать существовать до своего TTL;
- не являются единственным источником восстановления run;
- не влияют на сохранённые run/input copies.

## 19.4. Следствие

Удаление или очистка temporary files после запуска:
- не должно разрушать history;
- не должно делать run невоспроизводимым, если run input copies сохранены корректно.

---

## 20. Module-specific file contracts

## 20.1. Wildberries

Один запуск WB принимает:
- ровно 1 файл цен
- от 1 до 20 файлов акций

Следовательно, active temporary set WB должен уметь хранить:
- один price file slot
- множественный promo files slot

При замене price file:
- в активном наборе должен остаться только новый price file

При замене/удалении promo files:
- active set обновляется соответственно.

## 20.2. Ozon

Один запуск Ozon принимает:
- ровно 1 файл .xlsx

Следовательно, active temporary set Ozon должен фактически быть однофайловым.
Новая загрузка файла Ozon:
- заменяет предыдущий активный файл этого же user+store+module набора.

## 21. Правила журналирования файловых событий

Обязательные события, связанные с файлами:

- file_uploaded
- temporary_file_deleted
- temporary_file_replaced
- result_downloaded
- source_file_downloaded
- temporary_files_auto_purged
- old_result_removed_on_new_success
- system_error

## 21.1. Что должно логироваться при upload

Минимум:
- user_id
- store_id
- module_code
- original_filename

## 21.2. Что должно логироваться при replace/delete

Минимум:
- user_id
- store_id
- module_code
- file identity

## 21.3. Что должно логироваться при superseded

Минимум:
- old run id
- new run id
- store_id

- reason = superseded

## 21.4. Что должно логироваться при retention delete

Минимум:
- file metadata id
- storage path
- reason
- timestamp
- scope of cleanup job

## 22. Обязательные инварианты файловой архитектуры

1. Temporary uploads и run files не смешиваются.
2. Каждый run имеет собственные физические input copies.
3. Между разными runs дедупликация input files запрещена.
4. Check не создаёт output file.
5. Successful process создаёт output file.
6. Failed/validation_failed process не должен оставлять доступный output result.
7. Старый successful result при новом successful result становится superseded.
8. Historical run сохраняется независимо от доступности файлов.
9. Недоступность файла должна быть видна в UI.
10. Прямой публичный доступ к файлам запрещён.
11. Temporary files очищаются автоматически через 24 часа.
12. После очистки временного набора экран должен считать набор пустым.
